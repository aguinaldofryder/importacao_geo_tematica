## User Story

> **Como** operador de importação, **quero** rodar `importacao-geo mapear --arquivo PLANILHA.xlsx --fluxo {territorial|predial} [--saida CAMINHO]`, **para que** o sistema valide a conexão com o banco, leia os cabeçalhos da planilha, classifique colunas (Fase 1), execute o auto-mapeamento contra o catálogo `campo`/`alternativa` (Fase 2) e grave um `mapping.json` inicial — pronto para que eu edite manualmente apenas os itens marcados como `PENDENTE` antes de chamar `validar` e `importar`.

| AC3 | Pipeline de execução em **6 etapas sequenciais** (cada falha encerra com exit ≠ 0 e mensagem PT específica via `spec.commandLine().getErr()`): (1) **Validação de conexão fail-fast** — abre `Connection` via `Instance<DataSource>` (mesmo padrão de `ValidarConexaoCommand`) e executa `SELECT 1`; em caso de falha, mensagem `"✗ Falha de conexão: <mensagemAmigavel>"` (reusa `ValidarConexaoCommand.mensagemAmigavel(Exception)` — método já é package-protected `static`, basta exposição via `import` no mesmo pacote `cli`). (2) **Validação do arquivo** — `Files.exists(--arquivo)` e extensão `.xlsx` (ambas já validadas por `ExcelLeitor.abrir`, mas a mensagem é re-emitida pelo orquestrador como `"✗ Arquivo inválido: <causa>"`). (3) **Carga do catálogo** — `campos = campoRepository.listarPorFluxo(fluxo)`; lista vazia é considerada erro de configuração (`"✗ Catálogo vazio para fluxo <fluxo>: verifique a tabela campo/grupocampo"`) — exit 1. (4) **Abertura da planilha + Fase 1** — `try (ExcelSessao sessao = ExcelLeitor.abrir(arquivo))`; obtém `headers = sessao.cabecalhos()`; instancia `ClassificadorColunas(config.mapeamento().caseSensitive(), config.mapeamento().trimEspacos())`; chama `classificar(headers, codigoImovelConfig.por(fluxo), colunasFixasConfig.por(fluxo))`. (5) **Acumulação de DISTINCT por header `MULTIPLA_ESCOLHA`** — varre `sessao.linhas()` **uma única vez** (NFR-04, stream-once garantido pela 2.1 AC2); para cada header em `classificacao.dinamicas()` cujo `Campo` correspondente seja `MULTIPLA_ESCOLHA`, acumula `Set<String>` dos valores distintos não-vazios em um `Map<String,Set<String>>` interno. (6) **Auto-mapeamento + persistência** — monta `EntradaAutoMapeamento` (com callbacks `alternativaRepository::listarPorCampo` e `header -> distinctMap.getOrDefault(header, Set.of())`), invoca `autoMapeador.mapear(entrada)`, e grava via `mapeamentoStore.salvar(mapeamento, --saida)`. |
| AC4 | **Resolução do nome do arquivo da planilha** (consumido por `EntradaAutoMapeamento.nomeArquivoPlanilha`, conforme Story 3.2 AC2): o orquestrador extrai apenas o nome do arquivo via `arquivo.getFileName().toString()` (não o `Path` completo) — alinha ao exemplo canônico §5 linha 112 (`"TABELA_TERRITORIAL_V001.xlsx"`). Esta é a responsabilidade de wiring delegada ao 3.4 pela Story 3.2. |
| AC5 | **Exit codes** (alinhados a FR-16 — interpretado para o subcomando `mapear`): `0` em sucesso completo (independente do número de itens `PENDENTE` no `mapping.json` — ter pendentes é o **comportamento esperado** desta fase, não um erro; o gate de pendentes vive na 3.5/4.5). `1` em qualquer falha de infra antes de produzir o `mapping.json`: conexão DB, arquivo ausente/inválido, catálogo vazio, erro de I/O ao gravar, exceção inesperada não-coberta. **Não há exit `2`** nesta story (esse exit code é da Story 4.5 — execução com erros por linha). |
| AC6 | **Relatório final ao stdout (sucesso)** — formato em **3 blocos** terminados por linha em branco: (a) **resumo numérico** em uma linha — `"✓ Mapeamento concluído: <T> coluna(s) dinâmica(s) — <M> mapeada(s), <P> pendente(s). ## Dev Notes

### Decisões arquiteturais desta story

- **`MapearCommand` é orquestrador puro, não contém lógica de domínio.** A 3.4 é deliberadamente "thin": valida conexão (delegando à mesma `mensagemAmigavel` da 1.4), abre planilha (delegando ao `ExcelLeitor` 2.1), classifica (`ClassificadorColunas` 2.4), auto-mapeia (`AutoMapeador` 3.2) e persiste (`MapeamentoStore` 3.1). A novidade é apenas o **wiring** — incluindo o segundo passe `MULTIPLA_ESCOLHA` que a 3.2 explicitamente delegou ao orquestrador (Story 3.2 Dev Notes "trade-off"). Manter o command thin facilita: (a) testes E2E direcionados aos contratos (não à lógica), (b) reuso futuro do mesmo wiring por uma API web caso surja, (c) cobertura JaCoCo concentrada nos módulos críticos NFR-09 que já estão acima de 80 %.

- **Por que `Callable<Integer>` em vez de `Runnable`** (AC2). A `Runnable` original do stub 1.3 não permite retornar exit code; picocli encerra com 0 sempre. FR-16 e AC5 exigem distinção entre sucesso (0) e falha de infra (1). Migrar é trivial (1 linha de assinatura + `return Integer`); padrão já em uso por `ValidarConexaoCommand`.

- **Por que validar conexão antes de abrir planilha** (AC3 etapa 1). Fail-fast, alinhado FR-02. Razões cumulativas: (a) abrir e parsear `.xlsx` de 100k linhas custa ~2-5 s; descobrir falha de credencial só ao final é UX ruim; (b) se o operador esqueceu `./config/application.properties`, melhor falhar imediatamente; (c) catálogo precisa ser carregado (etapa 3) antes da Fase 2, então validar conexão antes evita retry semi-completo.

- **Por que `--saida` recebe `Path` literal e não respeita `importacao.saida.diretorio`**. A config `ImportacaoConfig.Saida.diretorio` (`./saida`) é para os artefatos de **importação** (`.sql` + `.log` da Story 4.x), não para o `mapping.json` que vive **antes** da importação. O `mapping.json` é input do operador (FR-08 — edição manual), naturalmente fica em diretório de trabalho do projeto. Default `./mapping.json` é o esperado; permitir override via `--saida` cobre cenários de pipelines automatizados que querem nomear o arquivo. **Não aplicar `sufixoTimestamp`** ao `mapping.json` (anti-pattern: o usuário não quer descobrir qual `mapping-2026-04-28T12-15-00.json` é o que ele acabou de editar).

- **Por que extrair apenas o `getFileName()` para `nomeArquivoPlanilha`** (AC4). Story 3.2 AC2 fixou esse contrato. Razão: o `mapping.json` pode ser comitado em VCS (FR-08 — edição manual; revisão DBA), e o caminho absoluto na máquina do operador (`/home/maria/Trabalho/import/X.xlsx`) é ruído + leak de info. Apenas o nome (`X.xlsx`) é semanticamente relevante.

- **Por que skippable o segundo passe DISTINCT** (AC13). Em planilhas territoriais reais, predominam tipos `TEXTO`/`DECIMAL`; `MULTIPLA_ESCOLHA` aparece apenas em colunas com domínio fixo (ex.: `TIPO_MURO`, `PADRAO_CONSTRUTIVO`). Se o catálogo do fluxo não tem **nenhum** `MULTIPLA_ESCOLHA` casável, a varredura `sessao.linhas()` é puro custo (potencialmente segundos). Evitar economiza tempo + memória sem perda funcional (mapa vazio passa para callback que retorna `Set.of()` para qualquer header). NFR-04 (streaming) é respeitado tanto com quanto sem essa otimização — esta story apenas a aplica deliberadamente como decisão de UX.

- **Por que `LinkedHashSet` para acumular DISTINCT** (AC14). Story 3.2 AC12 exige determinismo na ordem das chaves de `ColunaDinamica.alternativas` "na ordem em que os valores DISTINCT chegam do callback". `HashSet` quebraria essa garantia (ordem indefinida); `LinkedHashSet` preserva ordem de inserção mantendo `O(1)` para teste de duplicidade.

- **Por que `CodigoImovelConfig` separada de `ImportacaoConfig`**. Mesma razão arquitetural da `ColunasFixasConfig` (Story 2.4): `ImportacaoConfig` é raiz com `prefix=importacao` e dois sub-grupos (`saida`, `mapeamento`); adicionar `codigoImovel` como terceiro sub-grupo aumenta acoplamento (uma única interface conhece tudo). Manter como `@ConfigMapping` separado com `prefix=importacao.codigo-imovel` mantém SRP e facilita futuro: outras configurações por fluxo (ex.: `tabela-codigo`, `coluna-codigo-fisica` da 4.2) podem ganhar suas próprias interfaces sem inflar `ImportacaoConfig`.

- **Por que default `"MATRICULA"` em vez de obrigar a propriedade**. Fixture canônica `mapping-exemplo-territorial.json` (Story 3.1) e exemplo §5 da Arquitetura usam `"MATRICULA"` literal. Para o município de Paranacity (banco em `dump-paranacity.backup`), `MATRICULA` é o nome efetivo. Tornar a propriedade **opcional com default** = caso comum funciona out-of-the-box; municípios com nomenclatura diferente sobrescrevem em `./config/application.properties`. **Validação ainda ocorre** no `ClassificadorColunas` (Story 2.4 AC5) — se a planilha não tiver coluna com aquele nome, lança `ColunaCodigoAusenteException` PT.

- **Por que reusar `ValidarConexaoCommand.mensagemAmigavel`** (AC3 etapa 1, AC7). Mantém um único catálogo PT de mapeamento `SQLState`→mensagem; evita que duas mensagens divirjam se o catálogo evoluir (ex.: novo `SQLState` adicionado). O método já é `static` package-private — visível dentro do pacote `cli`. Alternativa "duplicar": rejeitada (DRY + risco de drift); alternativa "extrair para `cli.SqlStatePtMessages`": viável mas overkill para esta story (1 caller adicional). Caso surja terceiro caller, refatorar.

- **Por que teste E2E `@QuarkusTest` em vez de unitário do `MapearCommand`**. O command é wiring puro: mockar `Instance<DataSource>`, `CampoRepository`, `MapeamentoStore` etc. produziria teste tautológico que verifica chamadas Mockito, não comportamento. O valor está em verificar que **o pipeline real** funciona ponta-a-ponta com banco real (Testcontainers) e planilha real. Padrão idêntico a `ValidarConexaoCommandTest` da 1.4 (que também é E2E `@QuarkusTest` + Testcontainers, não unitário). Custo ~3-5 s por teste (start container + boot Quarkus); 6 cenários ~20-30 s — aceitável.
