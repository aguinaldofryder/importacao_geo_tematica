## User Story
> **Como** equipe, **quero** conseguir compilar e executar o binário nativo em dev local (Linux), **para** validar o suporte completo a `native-image` — cobrindo reflexão, recursos e drivers — **antes** de ativar o CI nativo (Story 6.2).

## Contexto

O projeto está funcionando em modo JVM (Épicos 1–5 completos, 424 testes verdes). A Story 4.5
implementa o `ImportarCommand` completo; esta story assume esse pré-requisito atendido e acrescenta
apenas o suporte de compilação nativa e a documentação de como mantê-lo.

O principal risco técnico do projeto — **R1** (Arquitetura §10) — é exatamente a reflexão de
Jackson e do `fastexcel-reader` no binário nativo. Esta story elimina esse risco de forma
controlada: primeiro executa o `native-image-agent` em JVM para capturar o que é acessado via
reflexão em tempo de execução, depois gera os arquivos de configuração necessários e confirma
o binário funciona nos cenários reais.

**Estado dos registros de reflexão ao início desta story:**

| Arquivo | Tipos cobertos |
|---|---|
| `DominioReflection` | `Campo`, `Alternativa`, `GrupoCampo` |
| `MapeamentoReflection` | `Classificacao`, `Mapeamento`, `ColunaDinamica`, `EntradaAutoMapeamento` |
| `ResumoSnapshot` | anotação direta `@RegisterForReflection` |
| `META-INF/native-image/` | **não existe** — criado por esta story |

O perfil `-Pnative` no `pom.xml` já existe (adicionado pelo scaffold Quarkus, Arquitetura §9.1).

## Dev Notes

### Pré-requisitos de ambiente

- **GraalVM JDK 21** com `native-image` no PATH. A versão exata deve casar com a que será
  configurada no CI (Story 6.2) para evitar divergências de suporte a reflexão.
  Recomendado: `sdk install java 21.0.x-graalce` via SDKMAN, ou download GraalVM CE 21.
- A variável `JAVA_HOME` deve apontar para o GraalVM JDK (não para o Temurin do CI).
  Verificar: `java -version` deve reportar `GraalVM CE`.
- Docker em execução (banco para T3 e T8).

### Perfil `native` no pom.xml

O perfil já existe — nenhuma alteração esperada. Se T1 revelar necessidade de flags adicionais
ao compilador nativo (ex.: `--initialize-at-run-time`, `--trace-class-initialization`), adicioná-las
via `quarkus.native.additional-build-args` no `application.properties` ou no `pom.xml`:

```xml
<quarkus.native.additional-build-args>
  --initialize-at-run-time=com.exemplo.ClasseProblema
</quarkus.native.additional-build-args>
```

### Como rodar o `native-image-agent`

```bash
# 1. Build do JAR executável (JVM)
./mvnw package -DskipTests

# 2. Primeira execução — cria o diretório de saída
mkdir -p target/native-image-agent
java -agentlib:native-image-agent=config-output-dir=target/native-image-agent \
     -jar target/importacao-geo-*-runner.jar \
     validar-conexao

# 3. Execuções subsequentes — ACUMULA com config-merge-dir
java -agentlib:native-image-agent=config-merge-dir=target/native-image-agent \
     -jar target/importacao-geo-*-runner.jar \
     mapear --arquivo src/test/resources/fixtures/TABELA_TERRITORIAL_V001.xlsx \
            --fluxo territorial

java -agentlib:native-image-agent=config-merge-dir=target/native-image-agent \
     -jar target/importacao-geo-*-runner.jar \
     validar --mapeamento mapping.json

java -agentlib:native-image-agent=config-merge-dir=target/native-image-agent \
     -jar target/importacao-geo-*-runner.jar \
     importar --arquivo src/test/resources/fixtures/TABELA_TERRITORIAL_V001.xlsx \
              --fluxo territorial --mapeamento mapping.json

# 4. Inspecionar saída
ls target/native-image-agent/
# reflect-config.json  resource-config.json  proxy-config.json  serialization-config.json
```

> **Atenção ao triar**: o agente captura reflexão do próprio JDK (`java.lang.*`,
> `java.util.*`, `sun.*`) e de todo o Quarkus. Essas entradas **não devem** ser copiadas
> para o `reflect-config.json` do projeto — GraalVM e as extensões Quarkus já as cobrem.
> Filtrar por namespace: manter apenas `org.dhatim.*` (fastexcel), `com.fasterxml.jackson.*`
> que não sejam cobertas por `quarkus-jackson`, e qualquer classe do projeto que apareça
> sem `@RegisterForReflection`.

### fastexcel-reader e OOXML

O `fastexcel-reader` foi escolhido por ser amigável a `native-image` (Arquitetura §2 —
justificativa explícita). Na prática, isso significa que ele evita reflexão dinâmica na leitura
do OOXML. Porém, pode requerer entradas em `resource-config.json` para incluir schemas XML
embutidos no JAR (ex.: `*.xsd`, `*.xml` dentro do JAR de fastexcel). O agente capturará isso
em `resource-config.json` com padrão `"pattern": "\\Qorg/dhatim/..."`. Incluir conforme necessário.

### Localização do `META-INF/native-image/`

Conforme AGENTS.md e CONVENCOES.md §Native Image:

```
src/main/resources/META-INF/native-image/
└── reflect-config.json          (entradas não cobertas por @RegisterForReflection)
└── resource-config.json         (se necessário)
```

O Quarkus processa automaticamente esse diretório durante o build nativo.

### Medição de startup e tamanho

```bash
# Tamanho
ls -lh target/importacao-geo-*-runner

# Startup (medir 3 vezes, usar mediana)
time ./target/importacao-geo-*-runner validar-conexao
```

Valores esperados conforme Arquitetura §9.3: binário 60–90 MB; startup < 200 ms em hardware
moderno (limite NFR-03 é 500 ms). Se o binário ultrapassar 90 MB, investigar com
`native-image-inspect` antes de considerar `upx` (ver Arquitetura §10).
