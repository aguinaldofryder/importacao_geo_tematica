## User Story
> **Como** equipe, **quero** uma suite E2E que execute o **binário nativo** contra um banco real, **para** travar regressões de reflexão/native antes de qualquer release.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | Script `scripts/e2e.sh` (Linux) e `scripts/e2e.ps1` (Windows) executam fluxo completo: sobe PostgreSQL Docker → seed → `validar-conexao` → `mapear` contra amostra → edita `mapping.json` (preenche pendências previsíveis) → `importar` |
| AC2 | Amostras fixture: 10 linhas de cada planilha em `src/test/resources/fixtures/TERRITORIAL-10linhas.xlsx` e `PREDIAL-10linhas.xlsx` |
| AC3 | Seed SQL do catálogo casado com as fixtures em `src/test/resources/fixtures/seed-catalogo.sql` |
| AC4 | Assertiva 1: `.sql` gerado é sintaticamente válido (`psql --single-transaction --set ON_ERROR_STOP=on` em dry-run, ou equivalente) |
| AC5 | Assertiva 2: resumo contém contadores esperados (ex.: 10 lidos, 10 sucesso, 10 UPDATEs, N respostas) |
| AC6 | Assertiva 3: `grep -i "insert into tribcadastro"` no `.sql` retorna 0 linhas (CON-03 validada fim-a-fim) |
| AC7 | Integrado ao workflow CI como job dependente de `6.2` — falha do E2E **bloqueia** a tag de release |
| AC8 | Executa em ambos SOs (Linux e Windows) — matrix |

## Tasks

## Dev Notes

- Usar `psql -h localhost -U postgres -f /tmp/saida.sql --set ON_ERROR_STOP=on` para dry-run sem commit (iniciar transação e ROLLBACK ao final).
- Amostras devem ser **sintéticas** (não dados reais da prefeitura) — qualquer PII ou matrícula real é proibida.
- Se o teste demora > 5 min, paralelizar territorial e predial.
