## User Story
> **Como** equipe, **quero** uma suite E2E que execute o **binário nativo** contra um banco real, **para** travar regressões de reflexão/native antes de qualquer release.

## Dev Notes

### Estratégia geral dos scripts

Os scripts recebem o caminho do binário nativo como `$1` / `$Args[0]`, sem valor padrão — falha explícita se não fornecido. O banco é levantado via `docker run`, o schema é criado via seed, e ao final o container é removido via `trap` (mesmo em caso de falha). Cada fase é prefixada com `[E2E] <fase>` no stderr para facilitar diagnóstico no CI.

### Fases do `e2e.sh`

```
Fase 0 — Pré-requisitos: verificar docker, psql, jq no PATH; falhar com mensagem clara se ausentes.
Fase 1 — Subir PostgreSQL: docker run --rm -d --name e2e-pg -p 5432:5432 -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=paranacity postgres:16
Fase 2 — Aguardar prontidão: loop de até 30 tentativas (sleep 1s) executando `docker exec e2e-pg pg_isready -U postgres`
Fase 3 — Seed: psql -h 127.0.0.1 -U postgres -d paranacity -f src/test/resources/fixtures/seed-e2e.sql
Fase 4 — validar-conexao: "$BINARY" validar-conexao; esperar exit 0
Fase 5 — Fluxo TERRITORIAL: cd em diretório de trabalho temporário; "$BINARY" mapear --arquivo .../TERRITORIAL-10linhas.xlsx --fluxo territorial; "$BINARY" importar --arquivo .../TERRITORIAL-10linhas.xlsx --fluxo territorial --mapeamento ./mapping.json; capturar caminho do .sql e .log gerados
Fase 6 — Fluxo PREDIAL: análogo à fase 5 com --fluxo predial em diretório separado
Fase 7 — Assertivas: AC4 (sintaxe SQL), AC5 (contadores JSON), AC6 (ausência INSERT proibido)
Fase 8 — Cleanup: trap executa `docker rm -f e2e-pg`
```

### Estratégia de diretório de trabalho

Cada fluxo executa em subdiretório próprio (`/tmp/e2e-territorial-$$` e `/tmp/e2e-predial-$$`) para evitar colisão de nomes de arquivo `.sql` / `.log` / `mapping.json`. O binário lê `./mapping.json` e escreve em `./saida/` por default (conforme `application.properties`); rodar em diretório temporário garante isolamento sem precisar passar flags extras de saída.

### Seed E2E (`seed-e2e.sql`)

O seed deve garantir que `mapear` produza **zero colunas PENDENTE** — caso contrário `importar` falha na validação do mapeamento. Para isso:

1. **Catálogo casado**: cada cabeçalho de coluna dinâmica nos fixtures xlsx deve aparecer em `campo.descricao` com `ativo = 'S'`; `campo.idgrupocampo` deve apontar para `grupocampo.funcionalidade = 'TERRENO'` (territorial) ou `'SEGMENTO'` (predial).
2. **Alternativas cobertas**: para colunas do tipo `MULTIPLA_ESCOLHA`, cada valor distinto usado nas 10 linhas fixture deve existir em `alternativa.descricao` para o `idcampo` correspondente.
3. **Linhas principais pré-existentes**: os 10 valores de `tribcadastrogeral_idkey` (territorial) e `idkey` (predial) usados nas fixtures devem existir nas tabelas `tribcadastroimobiliario` e `tribimobiliariosegmento`, respectivamente.
4. **Tabelas de respostas vazias**: `respostaterreno` e `respostasegmento` devem existir mas sem linhas — para exercitar tanto INSERT (novo) quanto UPDATE (re-execução, se o script rodar o importar duas vezes, o que pode ser adicionado como variante).

Usar o arquivo `src/test/resources/seed-catalogo.sql` existente como referência para a estrutura DDL; `seed-e2e.sql` pode reutilizar o DDL e acrescentar os DML necessários.

### Assertiva AC4 — Sintaxe SQL (dry-run com rollback)

```bash
# Envolve o .sql gerado em BEGIN/ROLLBACK para validação não-destrutiva
TMPFILE=$(mktemp /tmp/e2e-check-XXXXXX.sql)
{
  printf 'BEGIN;\n'
  cat "$SQL_FILE"
  printf '\nROLLBACK;\n'
} > "$TMPFILE"

psql -h 127.0.0.1 -U postgres -d paranacity \
     --set ON_ERROR_STOP=on -f "$TMPFILE"
# exit code non-zero → sintaxe inválida ou erro de execução
rm -f "$TMPFILE"
```

### Assertiva AC5 — Contadores via JSON do `.log`

O `ImportarCommand` anexa a linha JSON no formato (Arquitetura §7.2):
```json
{"evento":"resumo","fluxo":"territorial","lidos":10,"sucesso":10,"erro":0,"principal_updates":10,...}
```

Extração e verificação:
```bash
RESUMO=$(grep '"evento":"resumo"' "$LOG_FILE")
[ "$(echo "$RESUMO" | jq '.lidos')"            = "10" ] || fail "lidos != 10"
[ "$(echo "$RESUMO" | jq '.sucesso')"          = "10" ] || fail "sucesso != 10"
[ "$(echo "$RESUMO" | jq '.erro')"             =  "0" ] || fail "erro != 0"
[ "$(echo "$RESUMO" | jq '.principal_updates')" = "10" ] || fail "principal_updates != 10"
```

O `jq` é pré-instalado nos runners `ubuntu-latest` do GitHub Actions. Para uso local garantir via `apt install jq` / `brew install jq`.

### Assertiva AC6 — CON-03 fim-a-fim

```bash
if grep -qiE "insert into (tribcadastroimobiliario|tribimobiliariosegmento)" "$SQL_FILE"; then
  echo "[E2E] FALHA AC6: INSERT proibido encontrado em $SQL_FILE"
  exit 1
fi
```

Usar `-E` para expressão regular com alternância; testar com um `.sql` falso contendo INSERT para confirmar que a assertiva falha corretamente.

### Considerações Windows (`e2e.ps1`)

- `docker` deve estar no PATH (Docker Desktop com WSL2 backend é o padrão em `windows-latest` GH Actions).
- `psql` deve estar disponível; instalar via `choco install postgresql` ou incluir no PATH a instalação do `windows-latest` (PostgreSQL cliente está presente por padrão nos runners GH Actions).
- `jq` está pré-instalado em `windows-latest` GH Actions; para uso local: `winget install jqlang.jq`.
- Substituir `$(...)` por `$(...) | Out-String` onde necessário; usar `$LASTEXITCODE` em vez de `$?` para verificar exit codes de executáveis nativos.
- Paths com barras: usar `$PSScriptRoot` para caminhos relativos ao script; evitar hardcoded `/tmp/`, usar `$env:TEMP`.

### Integração CI — job `e2e` em `native-build.yml`

O job `e2e` deve depender do job `build` apenas na matriz `ubuntu-latest` (o binário Linux é suficiente para o teste de regressão E2E; o teste Windows pode ser adicionado na Story 6.6 se necessário, para não bloquear o release).

```yaml
e2e:
  needs: build          # aguarda ambos os jobs de matrix; na prática baixamos só o Linux
  runs-on: ubuntu-latest
  timeout-minutes: 10
  steps:
    - uses: actions/checkout@v4
    - uses: actions/download-artifact@v4
      with:
        name: importacao-geo-linux-x64
        path: dist/
    - name: Tornar binário executável
      run: chmod +x dist/importacao-geo-linux-x64
    - name: Executar suite E2E
      run: bash scripts/e2e.sh dist/importacao-geo-linux-x64
```

Nota: `needs: build` em um workflow com `build` como job de matrix aguarda **todos** os itens da matrix por padrão. Se apenas o Linux for necessário, refatorar o job de build para ter `id` separados por OS — avaliar junto com a Story 6.6.

### Dados sintéticos — requisitos mínimos das fixtures

- 10 linhas por fluxo; primeira coluna = código de imóvel (`tribcadastrogeral_idkey` ou `idkey`).
- Usar valores fictícios: `IMV-0001` … `IMV-0010` para territorial; `SEG-0001` … `SEG-0010` para predial — ou qualquer sequência numérica simples.
- Incluir pelo menos 1 coluna de cada tipo: `TEXTO`, `DECIMAL`, `DATA`, `MULTIPLA_ESCOLHA` (exercita todos os caminhos de coerção).
- Nenhum dado que se assemelhe a matrícula ou endereço real; sem CPF/CNPJ.

### Tempo estimado de execução do E2E

- Docker pull postgres:16: ~30 s (frio) / 0 s (cache).
- Seed + validar-conexao: ~5 s.
- mapear + importar × 2 fluxos: ~10 s (binário nativo inicia em ~7 ms, tempo dominado por I/O jdbc).
- Assertivas: ~2 s.
- **Total estimado: < 2 min** — bem dentro do timeout de 10 min do AC1.

## Testing

- Esta story **É** o teste — success criterion: suite E2E verde bloqueia/libera releases.
- Testar o caminho negativo: forçar uma linha com `tribcadastrogeral_idkey` inexistente → AC6 não afetado, mas `erro=1` aparece no JSON → adicionar assertiva de exit code `2` (AC5.2).
- Testar robustez: executar o script duas vezes consecutivas (re-run E2E) — segunda execução deve produzir exit 0 (idempotência dos UPDATEs e UPSERTs).
