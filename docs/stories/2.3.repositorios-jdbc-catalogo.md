## User Story
> **Como** sistema, **quero** repositórios que leiam o catálogo do PostgreSQL em modo read-only, **para** que o auto-mapeador tenha dados reais para casar.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | `CampoRepository.listarPorFluxo(Fluxo fluxo)` retorna `List<Campo>` via JOIN com `grupocampo`, filtrado por `grupocampo.funcionalidade = fluxo.funcionalidade()` e `campo.ativo = 'S'` |
| AC2 | `AlternativaRepository.listarPorCampo(long idCampo)` retorna `List<Alternativa>` |
| AC3 | `GrupoCampoRepository.listarPorFuncionalidade(String funcionalidade)` retorna `List<GrupoCampo>` |
| AC4 | Todos usam `AgroalDataSource` injetado |
| AC5 | SQL à mão com `PreparedStatement` (sem JPA/ORM) |
| AC6 | Classes marcadas como `@ApplicationScoped` |
| AC7 | Testes integração com Testcontainers + seed SQL (`src/test/resources/seed-catalogo.sql`) |
| AC8 | Invariante: nenhum método escreve no banco (teste verifica read-only por convenção) |

## Dev Notes

- Try-with-resources para `Connection`, `PreparedStatement`, `ResultSet`.
- JOIN típico:
  ```sql
  SELECT c.id, c.descricao, c.tipo, c.ativo, c.idgrupocampo
    FROM campo c JOIN grupocampo g ON c.idgrupocampo = g.id
   WHERE g.funcionalidade = ? AND c.ativo = 'S'
  ```
- Manter mapeamento `ResultSet → Campo` num método privado reutilizável.
