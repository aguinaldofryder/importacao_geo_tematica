## User Story

> **Como** sistema de importação, **quero** três repositórios JDBC read-only (`CampoRepository`, `AlternativaRepository`, `GrupoCampoRepository`) que carreguem o catálogo dinâmico do schema `aise`, **para que** o auto-mapeador (Story 3.2), o validador (3.3) e o gerador de SQL (4.5) tenham os IDs reais de `campo` e `alternativa` para casar headers de planilha e valores `MULTIPLA_ESCOLHA` — **sem** introduzir ORM, **sem** escrever no banco e **sem** depender do schema default `public`.

## Dev Notes

### Decisões arquiteturais desta story

- **Por que três repositórios separados (e não um `CatalogoRepository` único)** — alinhamento com o layout declarado em `docs/architecture/arquitetura.md` §8 (linha 240–241): pacote `catalogo/` lista explicitamente `CampoRepository`, `AlternativaRepository`, `GrupoCampoRepository`. Manter granularidade ajuda a paralelizar evolução (ex.: `AlternativaRepository` pode crescer com cache em story futura sem mexer nos demais).
- **JDBC puro com `PreparedStatement`, sem ORM** — decisão arquitetural já registrada em `AGENTS.md` §Stack: "Persistência: JDBC puro via `quarkus-agroal` + `quarkus-jdbc-postgresql`. **Sem ORM/Panache.**" Justificativas: (a) GraalVM `native-image` é frágil com Hibernate em comparação; (b) o domínio é um catálogo pequeno em 3 tabelas — overhead de ORM não compensa; (c) consistente com a Story 2.2 que escolheu `record` imutável em vez de `@Entity`.
- **`Fluxo` traduz para `funcionalidade` no chamador** — `CampoRepository.listarPorFluxo(Fluxo)` chama `fluxo.funcionalidade()` internamente, mas `GrupoCampoRepository.listarPorFuncionalidade(String)` aceita `String` cru. Razão: 2.2 deixou claro que `Fluxo` é vocabulário de **CLI** e `funcionalidade` é vocabulário de **catálogo**; um chamador que já tem o `String` do banco em mãos (ex.: diagnóstico, ferramentas) não deve ser obrigado a re-empacotar em `Fluxo`. A conversão fica do lado do código aplicacional, mantendo o repositório como adaptador fino.
- **`ImportacaoException` em vez de `SQLException` checada** — `SQLException` é checada (Java legacy) e poluiria toda a cadeia de chamadas. `ImportacaoException` (raiz `RuntimeException`, criada em 2.1) já é o canal de falhas de fluxo do projeto. Diferente de 2.2 (onde `IllegalArgumentException` foi usada para invariantes de construtor): aqui é falha de I/O **recuperável no nível do comando** — o comando CLI captura, loga em PT (NFR-01) e retorna exit code 1 (Arquitetura §7.3).
- **DDL no resource de teste, não em arquivo `schema.sql`** — `quarkus-flyway` não está no `pom.xml` e adicioná-lo só para testes seria custo desnecessário. DDL inline + `seed-catalogo.sql` separado mantém a setup auto-contida no `CatalogoPostgresResource`. Se a 2.4/4.x precisar de mais tabelas (`tribcadastroimobiliario` etc.), é **outra story** e provavelmente justificará Flyway — não antecipar aqui.
- **Subset reduzido das tabelas reais no DDL** — o banco de produção tem dezenas de colunas em `campo`/`alternativa`. O DDL de teste inclui **apenas** as colunas que os repositórios consultam (AC1–AC4). Adicionar colunas extras seria ruído e amarraria os testes ao schema completo do IPTU sem ganho. Quando a Story 4.x precisar de `tribcadastroimobiliario.tribcadastrogeral_idkey`, terá seu próprio resource ou estenderá este.
- **Construtor `@Inject` em vez de field injection** — alinhamento com práticas modernas Quarkus 3 + boa prática para native-image (campos `final` são amigáveis ao GraalVM). Também permite teste unitário direto (sem CDI) instanciando `new CampoRepository(mockDataSource)` se algum teste futuro quiser mockar.
- **Nome `CatalogoPostgresResource` (e não estender `cli.PostgresResource`)** — `cli.PostgresResource` é package-private e foca em conectividade pura no schema `public` (AC5 da 1.5 testa que senha não vaza no output, então é deliberadamente *minimalista*). O catálogo precisa de `aise` + DDL + seed — outra responsabilidade. Manter dois resources isolados respeita SRP e evita acoplamento entre stories.
- **Sufixo `Test` (não `IT`)** — Surefire roda `*Test.java`, Failsafe roda `*IT.java`. O projeto está com `skipITs=true` global (ver `AGENTS.md` §Comandos: `verify -Pnative` cita `skipITs=true por padrão`). Mesmo sendo testes integrados de fato, mantê-los como `Test` é a convenção do projeto e garante execução em todo `mvn verify`.

## Testing

- **Framework para testes integrados:** `@QuarkusTest` + `@QuarkusTestResource(value=CatalogoPostgresResource.class, restrictToAnnotatedClass=true)` + Testcontainers PostgreSQL 16-alpine.
- **Framework para teste de reflexão (AC7):** JUnit 5 puro (sem container, sem Quarkus context) — só inspeciona `Class.getDeclaredMethods()` dos 3 repositórios.
- **Localização produção:** `src/main/java/br/com/arxcode/tematica/geo/catalogo/`.
- **Localização testes:** `src/test/java/br/com/arxcode/tematica/geo/catalogo/`.
- **Localização seed:** `src/test/resources/seed-catalogo.sql` (acessado via classpath: `getClass().getResourceAsStream("/seed-catalogo.sql")`).
- **Convenção:** sufixo `Test` (Surefire), nunca `IT` (`skipITs=true` global — ver `AGENTS.md`).
- **Fixtures:** seed SQL com IDs explícitos para asserts determinísticos.
- **Cenários obrigatórios (AC11):**
  - `listarPorFluxo(TERRITORIAL)` → 3 campos (IDs 10, 11, 12 do seed; o ID 13 está `ativo='N'` e deve ficar fora).
  - `listarPorFluxo(PREDIAL)` → 2 campos (IDs 20, 21).
  - `listarTodos()` → 5 campos (todos exceto o `ativo='N'`).
  - `listarPorCampo(11)` → 2 alternativas (Alvenaria, Madeira).
  - `listarPorCampo(999_999)` → lista vazia, sem exceção.
  - `listarPorFuncionalidade("TERRENO")` → 1 grupo com `id=1`.
  - **Read-only invariant** (AC7) — reflexão sobre os 3 repositórios.
- **Cobertura esperada:** ≥ 90 % de linhas para os 3 repositórios (são adapters finos). O método privado `mapear` é coberto indiretamente pelos métodos públicos.
- **Critério NFR-09:** este pacote (`catalogo/`) **está** entre os críticos para a meta de 70 % global por ser dependência direta do auto-mapeador. Atingir ≥ 90 % é trivial dado o tamanho.
- **Performance:** `CatalogoPostgresResource` adiciona ~3–5 s de startup (Testcontainers). Container é estático e compartilhado entre testes da mesma classe — não inicializar novamente. `restrictToAnnotatedClass=true` evita penalizar classes que não precisam.

### Desvio em relação ao AC5 (resolvido)

A versão original do AC5 (v1.0) prescrevia injeção direta de `DataSource` por
construtor. A implementação usa `jakarta.enterprise.inject.Instance<DataSource>`
(lookup programático CDI) pelo mesmo motivo já adotado em
`cli.ValidarConexaoCommand:46`: com `%test.quarkus.datasource.active=false`
aplicado a todos os outros testes `@QuarkusTest` do projeto, a injeção direta
dispara `InactiveBeanException` durante o lifecycle de inicialização do CDI
(mesmo sem chamar nenhum método dos repositórios). A wrapper
`Instance<DataSource>` resolve o bean apenas quando `.get()` é invocado —
comportamento exigido pela estratégia de datasource opcional da Story 1.5.

**Status:** o AC5 foi emendado na v1.2 do Change Log para reconhecer
explicitamente ambas as formas de injeção (direta ou via `Instance<DataSource>`).
A implementação atual está conforme.
