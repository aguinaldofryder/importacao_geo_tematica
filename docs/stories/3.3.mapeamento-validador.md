## User Story

> **Como** sistema, **quero** recusar os comandos `validar` e `importar` se houver qualquer item `PENDENTE` ou qualquer inconsistência semântica no `mapping.json`, **para** impedir a geração de SQL com mapeamento incompleto ou corrompido por edição manual incorreta.

## Dev Notes

### Decisões arquiteturais desta story

- **Por que `ResultadoValidacao` impõe o invariante `valido ↔ pendencias.isEmpty()` no construtor** (não como validação separada). O invariante é uma pré-condição para o correto funcionamento de qualquer chamador: se `valido=true` mas `pendencias` não-vazia, o orquestrador pode prosseguir com SQL gerado incorretamente; o inverso (valido=false, pendencias vazia) deixaria o operador sem mensagem de erro acionável. Enforçar no construtor garante que nenhuma instância inválida possa existir — fail-rápido com mensagem PT específica. A alternativa (validação apenas nos chamadores) criaria invariante implícito difícil de rastrear.

- **Por que AC3 (alternativas null) se aplica somente a colunas MAPEADO** (não a PENDENTE). Após o `AutoMapeador` (Story 3.2), qualquer MULTIPLA_ESCOLHA com alternativa null resulta em `status=PENDENTE` (AC7 da 3.2) — AC2 já captura essa pendência. Aplicar AC3 também para PENDENTE geraria pendência duplicada para o mesmo header, confundindo o operador. O caso de proteção real de AC3 é contra edição manual (FR-08) que muda `status` de `PENDENTE` para `MAPEADO` sem preencher as alternativas null — nesse caso AC2 não dispara (status é MAPEADO), mas AC3 captura o null residual.

- **Por que AC5 (semântica de MAPEADO) são verificações adicionais além de AC2**. Mesmo lógica: proteção contra edição manual. Um `status=MAPEADO` sem `idcampo` ou sem `alternativas` em MULTIPLA_ESCOLHA é semanticamente inválido e causaria NullPointerException ou SQL incorreto no gerador (Story 4.2). Falhar aqui com mensagem PT clara é melhor que falhar silenciosamente durante a geração de SQL.

- **Por que `MapeamentoValidador` sem CDI**. Classe stateless sem dependências externas — nenhum `@Inject`, nenhuma config lida. Equivalente ao `AutoMapeador` (que recebe configuração via construtor) e `ClassificadorColunas` (mesma arquitetura). Testes JUnit puro são ~30× mais rápidos que `@QuarkusTest`. O orquestrador (Story 3.5 para o subcomando `validar`, Story 4.5 para o subcomando `importar`) instanciará com `new MapeamentoValidador()`.

- **Por que `ResultadoValidacao` não precisa de `@RegisterForReflection`**. O tipo não é serializado nem desserializado por Jackson — é um resultado interno computado em runtime e consumido pelo CLI para imprimir mensagens e definir exit code. Diferente de `Mapeamento`, `ColunaDinamica` e `EntradaAutoMapeamento` (que são escritos/lidos do `mapping.json`), `ResultadoValidacao` permanece inteiramente na heap da JVM durante a execução. A classe `MapeamentoReflection` não precisa ser atualizada por esta story.

- **Por que fail-all (não fail-fast)**. O operador edita o `mapping.json` manualmente (potencialmente com vários itens PENDENTE de uma vez — FR-08). Ver todos os problemas em uma única execução de `validar` é significativamente mais ergonômico do que corrigir um de cada vez. Padrão estabelecido nas Dev Notes da Story 3.3 stub e confirmado pelo mesmo raciocínio usado no `AutoMapeador` (AC7 da 3.2 retorna todos os itens PENDENTE de uma vez no JSON).

- **Ordem de verificação no passo dinâmicas**. Quando `status=PENDENTE`, os checks de AC5 (idcampo, alternativas) são **pulados** — seria redundante e confuso reportar "idcampo ausente" para uma coluna que o operador já sabe ser PENDENTE. A sequência é: primeiro checar `status`; se `PENDENTE` → emitir pendência AC2 e avançar para o próximo header; se `MAPEADO` → checar AC5 e AC3. Isso evita duplicação de pendências para o mesmo header.
