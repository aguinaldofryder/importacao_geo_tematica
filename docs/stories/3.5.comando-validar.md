## User Story

> **Como** operador de importação, **quero** rodar `importacao-geo validar [--mapeamento mapping.json]` **para que** o sistema verifique a completude e a consistência semântica do `mapping.json` (FR-07, FR-08, FR-09) **antes** de eu chamar `importar`, evitando que a Fase 3 produza SQL incorreto a partir de um mapeamento com itens `PENDENTE` ou corrompido por edição manual.

| AC1 | Subcomando `validar` registrado em `MainCommand` (já presente desde a Story 1.3) aceita as seguintes flags via picocli: `--mapeamento` / `-m` (opcional, `Path`, **default = `./mapping.json`**). Inclui `mixinStandardHelpOptions = true` (espelha `ValidarConexaoCommand` e `MapearCommand`). Help em PT (NFR-01) — `description` do `@Command` e da `@Option` em português. |
| AC2 | `ValidarCommand` migra de `Runnable` para `implements Callable<Integer>` (necessário para retornar exit codes — AC5). | AC3 | Pipeline de execução em **3 etapas sequenciais** (cada falha encerra com exit ≠ 0 e mensagem PT específica via `spec.commandLine().getErr()`): (1) **Validação do arquivo** — `Files.exists(--mapeamento)` falha → mensagem `"✗ Arquivo de mapeamento não encontrado: <caminho-absoluto>"`, exit 1; (2) **Carga e parsing** — invoca `mapeamentoStore.carregar(--mapeamento)`; qualquer `MapeamentoIoException` (JSON inválido, campo obrigatório ausente, valor enum inválido, sem permissão, etc.) → mensagem `"✗ Mapeamento inválido: <e.getMessage()>"`, exit 1 (não 2 — falha de mapping é categoria FR-16 "mapping inválido" = exit 1); (3) **Validação semântica** — invoca `mapeamentoValidador.validar(mapeamento)`; o resultado controla AC4 e AC5 (exit code + relatório). |
| AC4 | **Comando puramente offline** (NFR-04 lateral; alinha-se com a definição original do stub e §3 da Arquitetura): **não** abre conexão com o banco, **não** injeta `DataSource` nem `Instance<DataSource>`, **não** consulta repositórios (`CampoRepository`/`AlternativaRepository`). Toda a validação opera apenas sobre o conteúdo do `mapping.json` em memória. Útil em pipelines de revisão (CI/code review do `mapping.json` editado) onde o banco não está acessível. |
| AC5 | **Exit codes** (alinhados a FR-16 — interpretado para o subcomando `validar`): `0` em sucesso (`ResultadoValidacao.valido() == true`); `1` em qualquer falha — arquivo ausente, JSON inválido, ou `ResultadoValidacao.valido() == false` (qualquer pendência). **Não há exit `2`** nesta story (esse exit code é reservado para a Story 4.5 — execução com erros por linha durante a Fase 3, conceito que não se aplica ao `validar`). Falhas inesperadas (exceção não-coberta, fora do contrato `MapeamentoIoException` ou `IllegalArgumentException`) também retornam `1` com mensagem `"✗ Falha inesperada: <e.getMessage()>"` e `LOG.error` preservando stacktrace. |
| AC6 | **Relatório final ao stdout (sucesso)** — quando `ResultadoValidacao.valido() == true`, imprimir **uma única linha**: `"✓ Mapeamento completo. Pronto para importar."` (alinha exatamente o formato do stub original e mantém continuidade UX com o relatório de sucesso da 3.4). Exit 0. **Não** imprimir contadores adicionais (mapeados/fixas) — o `mapear` já fez essa contagem; o `validar` é um gate booleano. |
| AC7 | **Relatório final ao stderr (falha de validação semântica)** — quando `ResultadoValidacao.valido() == false`, imprimir em **3 blocos** terminados por linha em branco: (a) **cabeçalho** — `"✗ Mapeamento incompleto. <N> pendência(s) encontrada(s):"` onde `N = pendencias.size()`; (b) **lista agrupada por header** — para cada header com pelo menos uma pendência, imprimir `"  - <header>:"` seguido de `"      • <motivo>"` para cada motivo daquele header (motivos preservam o formato emitido pelo `MapeamentoValidador`: a pendência crua tem o prefixo `"Coluna 'X': "`; o orquestrador **strippa** o prefixo `"Coluna 'X': "` para evitar duplicação visual sob o `<header>:`); (c) **rodapé** — `"Edite o arquivo manualmente (FR-08) e rode 'validar' novamente: <caminho-absoluto>"`. Exit 1. |

## Dev Notes

### Decisões arquiteturais desta story

- **`ValidarCommand` é orquestrador thin — ainda mais thin que `MapearCommand`.** Pipeline de apenas 3 etapas (vs. 6 do `mapear`): validação de arquivo, parsing e validação semântica. Toda a lógica vive em componentes já testados (`MapeamentoStore` na 3.1, `MapeamentoValidador` na 3.3 com 18 testes JUnit puros). A 3.5 é predominantemente **wiring + UX**: agrupar pendências por header para melhor leitura humana é a única transformação não-trivial.

- **Por que comando puramente offline** (AC4). O `mapping.json` é o **artefato editável** do pipeline (FR-08). Cenários reais exigem rodar `validar` sem acesso ao banco: (a) revisão por DBA em ambiente onde o banco do município não está exposto; (b) CI que valida `mapping.json` em PR antes de qualquer deploy; (c) operador iterando local enquanto edita pendências. Forçar `DataSource` apenas para "manter paridade com `mapear`" seria gratuito e quebraria estes cenários. **Decisão simétrica futura**: `importar` (4.5) **precisa** do banco (gera SQL contra o schema real) — esse é o único comando além de `validar-conexao` e `mapear` que abre conexão; `validar` é o outlier consciente.

- **Por que `Callable<Integer>` em vez de `Runnable`** (AC2). Mesma razão de `MapearCommand` (Story 3.4 Dev Notes): `Runnable` não permite retornar exit code; FR-16 e AC5 exigem distinguir sucesso (0) de falha de validação (1). Migração trivial (1 linha de assinatura + `return Integer`).

- **Por que falha de mapping inválido = exit 1, não exit 2** (AC5). FR-16 categoriza explicitamente "mapping inválido" como exit 1 ("falha de configuração / conexão / mapping inválido"). Exit 2 é reservado para "execução completa com erros por linha" — conceito que **não existe** no `validar` (não há linhas executadas; há apenas validação estática do arquivo). A interpretação `validar incompleto = exit 1` mantém a semântica: o operador deve ser capaz de rodar `validar && importar` em sequência, e qualquer não-zero do `validar` deve abortar a chain.

- **Por que `--mapeamento` opcional com default `./mapping.json`** (AC1, AC12). Caso comum: operador acabou de rodar `mapear --saida ./mapping.json` (default do 3.4 AC1) e quer validar; tornar `--mapeamento` opcional permite `validar` sem repetir o caminho. Cenários de pipelines automatizados que usam outro nome (e.g. `mapping-territorial.json`) cobrem com `--mapeamento <path>` explícito. Default `./mapping.json` é exatamente o mesmo do `--saida` em `mapear` — simetria reduz superfície cognitiva.

- **Por que `MapeamentoValidador` como `static final`** (AC10). A Story 3.3 deliberadamente decidiu **não** anotar `MapeamentoValidador` com `@ApplicationScoped` (decisão alinhada ao `AutoMapeador` e `ClassificadorColunas`): classe stateless, sem dependências, instanciada com `new` pelos chamadores. No `ValidarCommand` (cujo escopo é `@Dependent` — uma instância por execução do comando), instanciar via `new` no `call()` cria uma instância nova a cada invocação — desperdício marginal, mas estilo desnecessário. `static final` resolve: instância única JVM-wide, thread-safe (validador é puro), lazy-initialization implícita pelo classloader. Mesma estratégia será aplicável ao `ImportarCommand` (Story 4.5).

- **Por que agrupar pendências por header no relatório** (AC7). O `MapeamentoValidador` (Story 3.3 AC6) emite cada pendência como `"Coluna 'X': motivo"`. Em mapeamentos reais, um mesmo header pode gerar **múltiplas** pendências (e.g. MULTIPLA_ESCOLHA MAPEADO com 5 alternativas null → 5 strings com prefixo `"Coluna 'X': alternativa 'Y' sem idAlternativa mapeado"`). Imprimir flat (uma string por linha) seria correto mas verbose; agrupar por header reduz ruído visual e ajuda o operador a identificar **quais colunas** precisam de atenção (não quais valores). Custo de implementação: 1 helper de regex + `LinkedHashMap` (~15 linhas).

- **Por que strippar o prefixo `"Coluna 'X': "` no relatório** (AC7). Se imprimíssemos `"  - X:\n      • Coluna 'X': motivo"`, o header `X` apareceria duplicado e o operador perderia tempo decifrando. Strippar o prefixo (via regex no helper de agrupamento) elimina a duplicação visual. Defesa em profundidade: se o `MapeamentoValidador` mudar o formato no futuro (improvável, é contrato AC6 da 3.3), a regex falha → header especial `"(sem header)"` mantém todas as mensagens completas; nenhuma informação se perde.

- **Por que reuso do `MapeamentoStore.carregar` em vez de parser próprio** (AC11). `MapeamentoStore.carregar` (Story 3.1) já trata 6 categorias de falha (`InvalidFormatException`, `ValueInstantiationException`, `MismatchedInputException`, `JsonParseException`, `AccessDeniedException`, `IOException`) com mensagens PT específicas. Re-implementar parsing no `ValidarCommand` duplicaria esse trabalho e arriscaria divergência. O `MapeamentoStore` é o **único ponto de entrada** para o JSON em todo o sistema — esse é o design da 3.1 e deve ser preservado.

- **Por que sem `Testcontainers` nos testes** (AC13). Comando puramente offline (AC4) → testar contra banco real seria contraditório e custaria 3-5s por teste em start de container. `@QuarkusTest` puro ainda boota o Quarkus (necessário para CDI: `@Inject MapeamentoStore` precisa do `ObjectMapper` configurado pelo `quarkus-jackson`) — custo ~1s amortizado. Cenários de fixture usam o **próprio `MapeamentoStore.salvar`** para escrever JSON válido ou JSON parcial em `@TempDir` — paridade total com produção, sem mock.
