## User Story

> **Como** operador de importação, **quero** que ao re-executar `mapear` sobre um `mapping.json` já existente, o sistema preserve o trabalho que já fiz (itens `MAPEADO` ou `PENDENTE` com `idcampo` preenchido manualmente), tente resolver automaticamente apenas os itens ainda incompletos (alternativas `null` em `MULTIPLA_ESCOLHA`), e indique claramente o que ficou pendente e por quê — **para que** eu não perca edições manuais e precise executar apenas um ciclo adicional para completar o mapeamento.

## Contexto e Lacunas

Esta story fecha duas lacunas identificadas após a implementação das Stories 3.2–3.4:

- **L1 — `mapear` ignora `mapping.json` pré-existente:** a implementação atual (Story 3.4) sempre gera um novo mapeamento do zero, descartando qualquer trabalho manual já realizado no arquivo de saída. Isso obriga o operador a re-editar manualmente os mesmos `idcampo` toda vez que re-executa o comando.
- **L4 — `MapeamentoValidador` não distingue subtipo de PENDENTE:** atualmente, qualquer coluna `PENDENTE` gera apenas a mensagem genérica `"Coluna '{header}': {motivo}"`. Não há como distinguir, por relatório ou programaticamente, se o item está pendente por "falta o `idcampo`" (bloqueador total) ou apenas por "falta resolver alternativas" (o operador já informou o `idcampo` e só falta o DBA fornecer os IDs de alternativas). Esta distinção permite re-mapeamento incremental e melhora a UX do relatório final do `mapear`.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | **Detecção de `mapping.json` pré-existente:** `MapearCommand.call()` verifica, antes de executar o pipeline completo, se o arquivo `--saida` já existe. Se existir, tenta carregá-lo via `mapeamentoStore.carregar(saida)`. Falha de leitura/parse nesta etapa imprime aviso PT em stderr (`"⚠ Aviso: mapping.json existente não pôde ser lido, iniciando mapeamento do zero: <causa>"`) e **continua** com o pipeline normal (comportamento degradado — não é falha fatal). Sucesso de leitura ativa o modo incremental descrito nos ACs seguintes. |
| AC2 | **Modo incremental — identificação de candidatos à resolução automática:** no modo incremental, o `MapearCommand` percorre as `colunasDinamicas` do `mapping.json` carregado e identifica os itens que satisfazem **todas** as condições: (a) `status == PENDENTE`; (b) `idcampo != null` (operador preencheu manualmente); (c) `tipo == MULTIPLA_ESCOLHA`; (d) `alternativas != null` e ao menos uma entrada tem valor `null` (há alternativas não resolvidas). Esses itens são chamados de **candidatos** internamente. |
| AC3 | **Modo incremental — varredura DISTINCT apenas para candidatos:** o segundo passe sobre `sessao.linhas()` (NFR-04 — stream-once) é executado **somente** se houver candidatos identificados em AC2. A varredura acumula `Set<String>` de valores distintos **apenas para os headers dos candidatos** (os mesmos headers presentes no `mapping.json` pré-existente como chaves em `colunasDinamicas`). Headers que não são candidatos não são acumulados. |
| AC4 | **Modo incremental — tentativa de resolução de alternativas:** para cada candidato, o `MapearCommand` invoca a lógica de `AutoMapeador.popularAlternativas` (seja via chamada direta ao método extraído para visibilidade `package-private` ou equivalente inline no command — decisão @dev) usando: `idcampo` já presente no candidato; valores distintos acumulados em AC3; `alternativaRepository.listarPorCampo(idcampo)` para obter as alternativas do banco. O resultado substitui o `ColunaDinamica` original do candidato no mapa in-memory. |
| AC5 | **Modo incremental — preservação do restante:** itens com `status == MAPEADO` **não são reprocessados** — são mantidos literalmente como estão no `mapping.json` carregado. Itens com `status == PENDENTE` que **não** são candidatos (sem `idcampo`, ou `tipo != MULTIPLA_ESCOLHA`, ou sem `alternativas` null) também são mantidos sem alteração. Colunas dinâmicas **novas** (presente na planilha atual mas ausente no `mapping.json` carregado) são mapeadas normalmente pelo pipeline de auto-mapeamento e adicionadas ao resultado. Colunas presentes no `mapping.json` carregado mas **ausentes** na planilha atual são **removidas** do resultado final (a planilha é a fonte de verdade dos headers). |
| AC6 | **Modo incremental — resolução parcial:** se após a tentativa de AC4 ainda houver entradas `null` em `alternativas`, o item permanece `PENDENTE` com `motivo` atualizado para `"<N> de <M> alternativas sem mapeamento (idcampo=<id> informado; verifique a tabela alternativa)"` — mantendo `idcampo` e `tipo` preenchidos para facilitar a próxima iteração manual. Os valores distintos da planilha que não tiveram match ficam como chaves com valor `null` no mapa `alternativas` (sugestão para o operador verificar grafias). |
| AC7 | **Relatório final — distinção de subtipo de PENDENTE:** o relatório impresso ao stdout por `MapearCommand` (atual bloco "(c) PENDENTES detalhados" — Story 3.4 AC6) deve distinguir dois subtipos: (a) `PENDENTE_SEM_CAMPO` — `idcampo == null`; prefixo `"[CAMPO PENDENTE]"`; (b) `PENDENTE_SEM_ALTERNATIVAS` — `idcampo != null`, `tipo == MULTIPLA_ESCOLHA`, ainda há `null` em `alternativas`; prefixo `"[ALTERNATIVAS PENDENTES]"`. Exemplo de linhas: `"  - [CAMPO PENDENTE] COLUNA_NOVA: Nenhum campo encontrado com descricao='COLUNA_NOVA'"` e `"  - [ALTERNATIVAS PENDENTES] TIPO_MURO: 2 de 4 alternativas sem mapeamento (idcampo=142 informado; verifique a tabela alternativa)"`. |
| AC8 | **`SubtipoPendente` enum — introdução opcional mas recomendada:** para suportar AC7 de forma limpa, introduzir `enum SubtipoPendente { SEM_CAMPO, SEM_ALTERNATIVAS }` em `br.com.arxcode.tematica.geo.mapeamento`. O `MapearCommand` usa esse enum internamente para classificar PENDENTES ao montar o relatório. **Alternativa aceita:** calcular o subtipo inline com `if (cd.idcampo() == null)` no próprio relatório sem introduzir o enum — decisão @dev. A escolha deve ser registrada nas Completion Notes. |
| AC9 | **`MapeamentoValidador` — mensagem de pendência distingue subtipo:** `MapeamentoValidador.validar` atualiza o formato da pendência para itens `PENDENTE` com `idcampo != null` e `tipo == MULTIPLA_ESCOLHA` e `alternativas` com ao menos um `null`. Para esses itens, a mensagem passa a ser `"Coluna '{header}': alternativas parcialmente resolvidas — verifique valores null em alternativas (idcampo={id})"`, diferenciando de itens bloqueados por ausência de `idcampo` (`"Coluna '{header}': {motivo}"`). Esta mudança é **backward-compatible** — itens sem `idcampo` continuam com a mensagem existente. |
| AC10 | **Modo inicial (sem `mapping.json` pré-existente):** quando o arquivo `--saida` não existe, o comportamento é **idêntico ao atual** (Story 3.4) — pipeline completo de classificação, auto-mapeamento e gravação. ## Dev Notes

### Decisões arquiteturais desta story

- **Por que detectar `mapping.json` pré-existente pelo caminho `--saida` (não por flag dedicada).** O operador executa `mapear --arquivo X.xlsx --saida mapping.json` repetidamente no mesmo diretório de trabalho. Adicionar `--incremental` ou `--merge` seria over-engineering: o comportamento correto é sempre "preservar o que existe, completar o que falta". A detecção silenciosa com aviso em caso de parse-error mantém UX mínima. Uma flag `--sobrescrever` pode ser adicionada em story futura se houver caso de uso legítimo para descartar trabalho manual.

- **Por que preservar itens `MAPEADO` sem reprocessar.** O operador pode ter corrigido manualmente um item que o auto-mapeador teria mapeado incorretamente (homonímia no catálogo, por exemplo). Reprocessar `MAPEADO` quebraria essa correção. O `mapping.json` é explicitamente um artefato de revisão humana (FR-08); uma vez marcado `MAPEADO`, a decisão é do operador.

- **Por que descartar headers da planilha ausentes no `mapping.json` carregado.** A planilha é a fonte de verdade. Se uma versão nova da planilha remove uma coluna, manter o mapeamento obsoleto no JSON produziria entradas mortas que confundiriam o `importar`. Se o operador precisar recuperar um mapeamento descartado, pode reaplicar manualmente.

- **Por que adicionar headers novos com auto-mapeamento e não com `PENDENTE` direto.** Novas colunas na planilha não devem exigir re-edição completa se o catálogo já as cobre. O auto-mapeamento tenta resolver; o que não resolver vai para `PENDENTE` normalmente — mesmo comportamento do modo inicial.

- **Por que L4 (`MapeamentoValidador`) é coberta nesta story e não em uma story separada.** A distinção de subtipo de PENDENTE é necessária tanto para o relatório do `mapear` (AC7) quanto para a mensagem do `validar` (via `MapeamentoValidador.validar`). As duas mudanças têm escopo pequeno e alto acoplamento; separá-las criaria uma story de 0,2 h com dependência circular. Implementar junto mantém coerência.

- **Por que não introduzir `StatusMapeamento.PENDENTE_ALTERNATIVAS` como novo valor de enum.** Mudar o enum `StatusMapeamento` quebraria o contrato de `MapeamentoStore` (deserialização de `mapping.json` existentes com valor não-mapeado pelo Jackson) e exigiria migrate de todos os arquivos `mapping.json` em uso. A abordagem adotada — distinguir subtipo **computado em runtime** a partir de `idcampo`+`tipo`+`alternativas` — é backward-compatible e não requer alteração de schema.

- **Reutilização vs. duplicação de `AutoMapeador.popularAlternativas` (T4).** O método é privado na Story 3.2. Alternativas: (a) expor como `package-private` — menos duplicação, mudança em classe já fechada; (b) replicar a lógica inline em `MapearCommand` — duplicação explícita, isolamento total. A Story 3.4 Dev Notes já registra a duplicação do helper `normalizar` como "debt explícito (ISSUE-3.4-02)". A decisão aqui agrava ou mitiga essa dívida. Recomendação: se `AutoMapeador.popularAlternativas` for exposto, abrir também `NormalizadorTextual` (ISSUE-3.4-02) na mesma story para liquidar o debt de uma vez; caso contrário, replicar inline com comentário `// TODO ISSUE-3.4-02`.

## Testing

- **Localização:** `src/test/java/br/com/arxcode/tematica/geo/cli/MapearCommandIncrementalTest.java` (novos) e `src/test/java/br/com/arxcode/tematica/geo/mapeamento/MapeamentoValidadorTest.java` (adição de cenários).
- **Convenção:** sufixo `Test` (Surefire); nada de `IT` (`skipITs=true` global — AGENTS.md).
- **Seed do catálogo:** `@BeforeEach` cria `grupocampo` (funcionalidade `TERRENO`), `campo` (id=10 `TIPO_MURO` MULTIPLA_ESCOLHA, id=11 `AREA_TERRENO` TEXTO), `alternativa` (Alvenaria→501, Madeira→502 para idcampo=10) no schema `aise` (via `PostgresResource` atualizado pela Story 3.4).

## Completion Notes

**Decisão T4 — reusar vs. replicar `popularAlternativas`:**
Optou-se por **reusar**, extraindo o núcleo lógico para o método privado `popularAlternativasComListas(long, Tipo, Set<String>, List<Alternativa>)`. O método original `popularAlternativas(String, Campo, EntradaAutoMapeamento)` delega para ele (package-private, uso interno). Novo método público `popularAlternativasIncremental(Integer, Set<String>, List<Alternativa>)` expõe a lógica para o `MapearCommand` sem duplicação. Isso mitiga parcialmente ISSUE-3.4-02 (duplicação do `normalizar`), que permanece como debt separado.

**Decisão T6 — enum `SubtipoPendente` vs. inline:**
Optou-se por **inline** com `if (cd.idcampo() == null)` no `imprimirRelatorio`. O enum separado não agrega valor para dois estados; a lógica é local ao método de relatório e ao `MapeamentoValidador`. AC8 aceita ambas as abordagens.

**RISCO-1 mitigado:** leitura do `mapping.json` e identificação de candidatos ocorrem **antes** de `ExcelLeitor.abrir()`. O segundo passe DISTINCT incremental está dentro do mesmo `try-with-resources` da sessão já aberta (mesmo bloco).

**RISCO-2 mitigado:** `mesclarMapeamento` constrói novo `LinkedHashMap` — não tenta modificar o `Collections.unmodifiableMap` retornado por `AutoMapeador.mapear`.

**GAP-1 mitigado:** fixture do cenário (b) usa `gerarPlanilhaBase` que inclui `AREA_TERRENO` entre os headers, garantindo que o campo esteja presente na planilha e não seja descartado como ausente antes de verificar a preservação de MAPEADO.
