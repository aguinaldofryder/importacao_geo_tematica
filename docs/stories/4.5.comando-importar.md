## User Story

> **Como** operador municipal, **quero** rodar `importacao-geo importar --arquivo X.xlsx --fluxo territorial --mapeamento mapping.json`, **para** gerar os artefatos `saida-territorial-{timestamp}.sql` e `.log` prontos para revisão e execução pelo DBA contra o banco de IPTU — sem jamais executar DML diretamente em produção.

## Dev Notes

### Decisões arquiteturais

- **Rastreamento de `houveErros` com array `boolean[] {false}` em lambda, em vez de método em `LogErros`.** `LogErros` não expõe flag de "houve erro" — seria necessário reabrir a Story 4.4 (fechada) ou adicionar um método que ela não prevê. Alternativas: (a) `LogErros.houveErros()` — clean, mas força reabertura de story fechada; (b) `AtomicBoolean` — thread-safe desnecessário (orquestrador é single-thread por design); (c) array `boolean[] houveErros = {false}` — idioma Java padrão para contador mutável em lambda, sem overhead; (d) record de resultado `Resultados(int sucesso, int erros)` retornado pela fase 5 — mais clean mas requer extração do processamento para método separado com return. **Decisão**: array `boolean[]` (opção c) — simples, sem mudanças em stories fechadas, consistente com `int[] linhaExcel`. Documentado em Javadoc do campo.

- **`ExistenciaRepository` em `catalogo/` vs. inline JDBC em `ImportarCommand`.** Mesclar a query de existência no `ImportarCommand` quebraria separação de responsabilidades (CLI + infra DB) e impediria testes isolados da query. `catalogo/` já contém `CampoRepository`, `AlternativaRepository`, `GrupoCampoRepository` — padrão estabelecido na Story 2.3. Custo: +1 bean CDI; benefício: testável, reutilizável (Story 6.3 pode reusá-lo), consistente com o layout de `docs/architecture/arquitetura.md:245`.

- **SQL com concatenação de tabela/coluna no `ExistenciaRepository`.** O nome da tabela e da coluna derivam de `fluxo.tabelaPrincipal()` / `fluxo.colunaChave()` — **valores controlados pelo enum `Fluxo`**, nunca do input do usuário ou do Excel. SQL injection contra esses valores é impossível; contudo, o parâmetro de valor (`codigo`) é obrigatoriamente parametrizado via `PreparedStatement.setString(1, codigo)`. Alternativa pré-computada: `Map<Fluxo, String>` de queries fixas — evitaria concatenação mas duplicaria informação que o enum já centraliza. Aceita a concatenação; documentada no Javadoc.

- **`int[] linhaExcel = {1}` para numeração de linha em lambda.** `Stream.forEach` exige efetivamente final no escopo externo. `int` primitivo não pode ser mutado. Opções: (a) array `int[] {1}` — idiomático, sem import extra; (b) `AtomicInteger` — overshoot para single-thread; (c) converter para `Iterator` com loop for-each — legível mas verboso. **Decisão**: (a) com comentário explicativo inline. Linha 1 = cabeçalho (já consumido por `ExcelSessao.cabecalhos()` no construtor); `linhaExcel[0]` incrementado **no início** de cada iteração, antes de qualquer processamento, garantindo que linha 2 seja o primeiro dado.

- **Validar arquivo antes de conexão (AC4 antes de AC3 Fase 2).** Descobrir que o arquivo `.xlsx` não existe após aguardar `acquisition-timeout=5s` de conexão seria frustrantemente lento. Validar primeiro (≈ instantâneo) permite fail-fast sem custo de I/O de rede. Exceção lógica: se o banco estivesse down, a mensagem sobre o `.xlsx` seria enganosa (o real problema é o banco). Contudo, a ordem "arquivo → banco" segue o princípio de "validar o que está sob controle do operador antes de validar dependências externas".

- **Reusar `ValidarConexaoCommand.mensagemAmigavel(Exception)`.** Esse método estático público em `ValidarConexaoCommand` formata `SQLException`s e outros erros de banco em mensagem PT sem stack trace. Reutilizá-lo em `ImportarCommand` garante consistência de UX entre os dois comandos. Não injetar `ValidarConexaoCommand` como bean — isso acoplaria subcomandos CLI.

- **`Coercionador`, `SqlGeradorUpdate`, `SqlGeradorUpsert`, `MapeamentoValidador` como `static final`.** Classes puras stateless sem CDI (decisões das Stories 3.3 e 4.1). `static final` evita instanciação desnecessária a cada `call()` (padrão estabelecido na Story 3.5 AC10 para `MapeamentoValidador`).

- **Erros de UPDATE e UPSERT registrados em chamadas separadas a `registrarErrosLinha` (AC8).** O `LogErros.registrarErrosLinha` recebe uma `List<String>` por chamada. Se uma linha produz erros de UPDATE e também de UPSERT, são duas chamadas com `resultado.erros()` de cada gerador. Isso garante que o log separa claramente "erros do UPDATE da tabela principal" de "erros de UPSERT das respostas" — facilita auditoria pelo operador. Alternativa (flat merge) perderia a separação por gerador. Custo: 2 entradas no log por linha com dupla falha (raro em prática).

- **Linha com código de imóvel vazio pula toda a linha, mesmo que outras colunas sejam válidas.** Sem o código, não é possível identificar qual registro atualizar — UPDATE e UPSERT perderiam a chave de referência. A linha deve ser pulada integralmente (consistente com "Código imobiliário ausente → log de erro, pula a linha inteira" do AGENTS.md §Domínio).

- **Arquivo `.sql` e `.log` com mesmo timestamp (AC11/AC12).** `LocalDateTime inicio` capturado uma única vez no início de `call()`. Garante que o par de artefatos tem nomes correlacionados, permitindo ao DBA referenciar os dois arquivos pelo mesmo sufixo. Usar `LocalDateTime` (não `OffsetDateTime`) no nome do arquivo por simplicidade do pattern `yyyyMMdd-HHmmss` (sem offset, que conteria `+` ou `-` problemáticos em nomes de arquivo no Windows).

- **`Files.createDirectories` para o diretório de saída.** O operador pode configurar `importacao.saida.diretorio=./saida/2026-05-03` sem criar subdiretórios manualmente. `createDirectories` é idempotente (não lança exceção se já existe). `IOException` durante a criação → stderr PT + exit 1 (falha de infra antes de gravar artefatos).

- **Exit `2` implica artefatos foram gravados.** Se a iteração completou mas houve erros por linha, ambos os artefatos (`.sql` e `.log`) são gravados antes de retornar exit 2. O operador pode inspecionar os dois arquivos para entender o que falhou e o que foi gerado com sucesso.

## Trade-offs registrados

- **Flag `boolean[] houveErros` vs. `LogErros.houveErros()`.** Custo: `ImportarCommand` carregar lógica de "houve erro" duplica marginalmente informação que `LogErros` já detém internamente. Benefício: evita reabrir Story 4.4 (fechada, auditada, 100%/100% de cobertura). Aceito para o MVP; reversível em 5.1 se necessário.
- **`ExistenciaRepository` em `catalogo/` vs. inline no comando.** Custo: +1 bean CDI, +1 arquivo de teste. Benefício: separação de responsabilidades, testabilidade, consistência com o layout `catalogo/` (2.3). Aceito sem ressalvas.
- **Seed via INSERT JDBC em `@BeforeEach` vs. dump de produção.** Dump (~346 MB) é impraticável em testes de unidade. INSERT mínimo é hermético, rápido e controlado. Custo: SQL de seed frágil se o schema da tabela principal mudar (acréscimo de NOT NULL sem default). Mitigação: INSERT mínimo com apenas os campos obrigatórios do schema; documentar no comentário do teste.
- **Exit `2` pressupõe artefatos gravados.** Implica que falha ao gravar arquivos após iteração bem-sucedida resulta em exit 1 (infra), não 2. Operador não terá artefatos em disco mas terá a mensagem de erro informando onde gravar falhou.
- **Linha com erro parcial (UPDATE ok, UPSERT com erro) não conta como sucesso.** Conservador; pode subestimar `registros.sucesso` em planilhas onde colunas dinâmicas têm valores não-mapeáveis mas as colunas fixas são válidas. Alternativa mais leniente: contar como sucesso se ao menos 1 SQL foi gerado. Mantida a opção conservadora por clareza de auditoria.
