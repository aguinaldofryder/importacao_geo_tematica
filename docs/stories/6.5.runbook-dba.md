## User Story
> **Como** DBA da prefeitura, **quero** um runbook passo-a-passo para aplicar o `.sql` gerado, **para** evitar corromper a base tributária de IPTU.

## Contexto

Com os Épicos 1–5 completos (implementação integral do CLI) e as Stories 6.1–6.3 entregues
(binário nativo, CI matrix, E2E), o produto está tecnicamente pronto para release.
O elo faltante é a documentação operacional para o DBA que receberá o arquivo `.sql` e precisará
aplicá-lo com segurança na base tributária de IPTU.

O DBA-alvo **não é desenvolvedor da ferramenta**: é um profissional de banco que conhece
PostgreSQL e o schema `aise`, mas não conhece a lógica interna do CLI. O `RUNBOOK-DBA.md`
é o único guia que ele terá para executar a aplicação em homologação e em produção.

**Contexto técnico crítico que o runbook deve transmitir:**

| Aspecto | Detalhe |
|---|---|
| O que o `.sql` contém | Apenas `UPDATE` em tabelas principais + `DELETE/INSERT` (UPSERT) em tabelas de respostas |
| O que o `.sql` **nunca** contém | `INSERT INTO tribcadastroimobiliario` ou `INSERT INTO tribimobiliariosegmento` |
| Tabelas afetadas — fluxo Territorial | `aise.tribcadastroimobiliario` (UPDATE) e `aise.respostaterreno` (DELETE+INSERT) |
| Tabelas afetadas — fluxo Predial | `aise.tribimobiliariosegmento` (UPDATE) e `aise.respostasegmento` (DELETE+INSERT) |
| UPSERT = DELETE+INSERT | O UPSERT de respostas exclui linhas existentes com o mesmo `(referencia, idcampo)` e as reinsere — não é in-place. O DBA deve considerar triggers de auditoria, se existirem. |
| Arquivo de referência | `.log` gerado junto com o `.sql` — contém linha JSON `"evento":"resumo"` com todos os contadores |
| Nomeação dos artefatos | `saida-{fluxo}-{timestamp}.sql` e `saida-{fluxo}-{timestamp}.log` |

**Fontes de verdade para o conteúdo do runbook:**

| Seção do runbook | Fonte |
|---|---|
| Checklist pré-aplicação | CON-03 (constraint de segurança do domínio); Arquitetura §4 |
| Comandos `psql` | `scripts/e2e.sh`; seed de testes `src/test/resources/fixtures/seed-e2e.sql` |
| Tabelas e schema | Arquitetura §5 (modelo de dados); AGENTS.md §Domínio |
| Counters do `.log` | Arquitetura §7.2 e §7.3 |
| Risco R4 | Arquitetura §10 — mitigação principal é este runbook |

O runbook **não deve** duplicar o `MANUAL.md` (uso do CLI pelo operador — Story 6.4).
O operador de importação é quem aciona o CLI; o DBA é quem aplica o `.sql` resultante.

## Dev Notes

### Público-alvo do runbook

O DBA conhece PostgreSQL e `psql`, mas **não** conhece o código-fonte do CLI.
O runbook deve:
- Usar linguagem de banco: "execute a transação", "restaure o dump", "verifique o plano de execução se necessário".
- Evitar termos Java/Quarkus (`bean`, `CDI`, `logger`, `native image`).
- Ser autocontido: um DBA sem acesso ao repositório deve conseguir executar os procedimentos com apenas o `.sql`, o `.log` e este runbook.

### Comando `psql` de aplicação (AC3 e AC4)

```bash
psql \
  --host=<host> \
  --port=5432 \
  --username=<usuario> \
  --dbname=<banco> \
  --single-transaction \
  --set ON_ERROR_STOP=on \
  -f saida-territorial-20260420-143447.sql
```

> **Nota:** verificar no código do gerador (Story 4.2/4.3) se o `.sql` já inclui `BEGIN`/`COMMIT`
> explícitos. Se sim, `--single-transaction` é redundante mas inócuo. Se não, é obrigatório.
> O Dev Notes deve ser atualizado após verificação — T3 é responsável por confirmar.

### Comando `pg_dump` de backup (AC4)

```bash
pg_dump \
  --host=<host> \
  --port=5432 \
  --username=<usuario> \
  --format=custom \
  --file=backup-geo-pre-aplicacao-$(date +%Y%m%d-%H%M%S).dump \
  -t aise.tribcadastroimobiliario \
  -t aise.respostaterreno \
  -t aise.tribimobiliariosegmento \
  -t aise.respostasegmento \
  <banco>
```

```bash

  --host=<host> \
  --port=5432 \
  --username=<usuario> \
  --dbname=<banco> \
  --clean \
  --if-exists \
  backup-geo-pre-aplicacao-20260420-143000.dump
```

### Nota sobre UPSERT via DELETE+INSERT

O `.sql` implementa UPSERT nas tabelas de respostas como `DELETE ... WHERE referencia = X AND idcampo = Y` seguido de `INSERT`. Isso significa:

- **Linhas em `respostaterreno`/`respostasegmento`** com o mesmo `(referencia, idcampo)` da planilha são **substituídas**, não atualizadas in-place.
- Se existirem triggers de auditoria que registram `DELETE`, eles serão acionados — comportamento esperado e documentado.
- O `pg_dump` pré-aplicação captura o estado anterior completo, o que é suficiente para rollback total.
