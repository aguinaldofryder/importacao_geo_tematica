## User Story
> **Como** DBA da prefeitura, **quero** um runbook passo-a-passo para aplicar o `.sql` gerado, **para** evitar corromper a base tributária de IPTU.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | `RUNBOOK-DBA.md` na raiz do repositório |
| AC2 | Seção 1 — **Checklist pré-aplicação**: validar timestamp do arquivo, confirmar fluxo (territorial/predial), conferir tamanho razoável, **grep garantindo `0` ocorrências** de `INSERT INTO tribcadastro*` (validação de CON-03 do lado do DBA) |
| AC3 | Seção 2 — **Aplicação em homologação**: procedimento com `BEGIN`, aplicação via `psql --single-transaction --set ON_ERROR_STOP=on`, conferência amostral, `COMMIT` ou `ROLLBACK` |
| AC4 | Seção 3 — **Aplicação em produção**: agendamento de janela, backup pré-aplicação (`pg_dump` das tabelas envolvidas), transação, monitoramento, critério de rollback |
| AC5 | Seção 4 — **Queries de sanidade pós-aplicação**: contagem de registros atualizados, spot-check de amostras, comparação contra o resumo do `.log` |
| AC6 | Seção 5 — **Rollback**: restore a partir do `pg_dump` + notificação |
| AC7 | Seção 6 — **Contatos de escalonamento** (placeholder a ser preenchido pelo cliente) |
| AC8 | Revisado e aprovado por um DBA real antes de release |
| AC9 | Linkado do `MANUAL.md` (seção "Para o DBA aplicar") |
