## User Story
> **Como** equipe, **quero** que toda tag `vX.Y.Z` publique um GitHub Release com binários e documentação, **para** o operador baixar tudo de um só lugar.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | Workflow (novo ou estendendo `native-build.yml`) dispara em tag `v*.*.*` e publica GitHub Release |
| AC2 | Release **depende** de: (a) unit + integration tests verdes, (b) cobertura ≥ 70% em módulos críticos (NFR-09), (c) build nativo verde em ambos SOs (Story 6.2), (d) E2E verde em ambos SOs (Story 6.3) — gating conforme Arquitetura §13.3 |
| AC3 | Assets anexados: `importacao-geo-linux-x64`, `importacao-geo-windows-x64.exe`, `MANUAL.md`, `RUNBOOK-DBA.md`, `application.properties.exemplo` |
| AC4 | Release notes preenchidas a partir de `CHANGELOG.md` (se existir) ou template default com link para milestone |
| AC5 | Primeira release: `v1.0.0` — acontece quando Épico 6 fecha |
| AC6 | Semver respeitado: MAJOR = quebra, MINOR = feature, PATCH = bug |
| AC7 | Smoke test com tag de teste (`v0.0.1-rc`) antes da `v1.0.0` oficial para confirmar pipeline |
| AC8 | Configuração do repo exige branches protegidas e PR review antes de merge para `main` (reforça Story 1.1 AC9) |

## Dev Notes

- Job de release deve ter `permissions: contents: write`.
- `softprops/action-gh-release` lê `CHANGELOG.md` automaticamente se configurado.
- Versionamento das releases: usar tags anotadas (`git tag -a v1.0.0 -m "..."`).
