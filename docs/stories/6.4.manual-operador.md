## User Story
> **Como** operador de importação, **quero** um manual em português cobrindo todo o uso da ferramenta, **para** configurar, rodar e interpretar sem depender de ajuda externa.

## Contexto

Com os Épicos 1–5 completos (implementação integral do CLI) e as Stories 6.1–6.2 entregues
(binário nativo + CI matrix Linux/Windows), o produto está tecnicamente pronto para release.
O elo faltante antes da automação de release (Story 6.6) é a documentação de uso destinada ao
operador de TI municipal.

O operador-alvo **não é desenvolvedor**: é um técnico que receberá o binário executável e um
`config/application.properties` configurado, e precisará conduzir o fluxo completo de importação
sem intervenção da equipe de desenvolvimento. O `MANUAL.md` é o único artefato de documentação
de uso que ele terá disponível no repositório/pacote.

**Fontes de verdade para o conteúdo do manual:**

| Fluxo de comandos | Arquitetura §3 (fluxo CLI completo) |

O manual **não deve** duplicar o `DEVELOPING.md` (guia do desenvolvedor, produzido em Story 6.1).
Procedimentos de banco (aplicar o SQL gerado, rollback) são escopo da Story 6.5 (`RUNBOOK-DBA.md`).

O operador é técnico de TI municipal, não desenvolvedor. O manual deve evitar:
- Termos de desenvolvimento (`build`, `JAR`, `Maven`, `Git`, `classpath`).

### Exemplos de comandos para a Seção 3 (Arquitetura §3)

**Linux:**
```bash
# 1. Validar conexão
./importacao-geo-linux-x64 validar-conexao

# 2. Gerar mapeamento (fluxo territorial)
./importacao-geo-linux-x64 mapear \
  --arquivo TABELA_TERRITORIAL_V001.xlsx \
  --fluxo territorial

# 3. Validar mapeamento (após editar o mapping.json)
./importacao-geo-linux-x64 validar --mapeamento mapping.json

# 4. Executar importação
./importacao-geo-linux-x64 importar \
  --arquivo TABELA_TERRITORIAL_V001.xlsx \
  --fluxo territorial \
  --mapeamento mapping.json
```

**Windows:**
```cmd
importacao-geo-windows-x64.exe validar-conexao
importacao-geo-windows-x64.exe mapear --arquivo TABELA_TERRITORIAL_V001.xlsx --fluxo territorial
importacao-geo-windows-x64.exe validar --mapeamento mapping.json
importacao-geo-windows-x64.exe importar --arquivo TABELA_TERRITORIAL_V001.xlsx --fluxo territorial --mapeamento mapping.json
```

### Explicação do `mapping.json` para o operador (Seção 3.3)

O operador pode encontrar itens `PENDENTE` quando o nome de uma coluna da planilha não casa automaticamente com nenhum campo cadastrado no banco. Procedimento para resolver:

1. Abrir `mapping.json` em qualquer editor de texto.
2. Localizar blocos com `"status": "PENDENTE"` — o campo `"motivo"` explica por que não foi mapeado.
3. Consultar o DBA responsável (ver `RUNBOOK-DBA.md`) para obter o `idcampo` correto.
4. Preencher `"idcampo": <número>` e alterar `"status": "PENDENTE"` para `"status": "MAPEADO"`.
5. Executar `validar` novamente; somente após zero itens `PENDENTE` o `importar` prosseguirá.
