## User Story

> **Como** operador municipal, **quero** ver uma barra de progresso atualizada em tempo real durante a execução do `importar`, **para** saber que o processo está em andamento e estimar quando vai terminar — especialmente em planilhas com milhares de linhas.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | Antes de iniciar o streaming, `ImportarCommand` realiza uma passagem rápida pelo arquivo `.xlsx` para contar o total de linhas de dados (excluindo o cabeçalho); o total é armazenado em variável local `int totalLinhas`. |

## Dev Notes

### Decisões arquiteturais

1. **`BarraProgresso` não-CDI, construída em `call()`:** stateful por execução (igual a `LogErros`, `ResumoExecucao`). CDI `@Dependent` introduziria complexidade sem benefício. Padrão consistente com o restante do pipeline.

2. **Contagem prévia de linhas via segunda abertura do arquivo:** `ExcelLeitor.abrir()` + `s.linhas().count()` é O(n) em streaming — compatível com `fastexcel-reader`. Custo: segunda leitura do arquivo `.xlsx` antes da importação. Para planilhas de até ~50 MB / ~50.000 linhas, esse custo é aceitável (~1–2 s). Alternativa (campo de metadados do XLSX) exigiria acesso à API interna do fastexcel, com risco de quebra em native-image. Decisão: segunda leitura explícita.

3. **Atualização a cada linha vs. a cada N linhas:** para planilhas grandes (~50.000 linhas), uma atualização por linha pode gerar overhead de I/O em stderr. Porém, para o tamanho esperado (~1.000–5.000 linhas do contexto IPTU municipal), é negligenciável. Em v1, atualizar a cada linha. Se necessário, v2 pode adicionar `if (linhaAtual % 10 == 0 || linhaAtual == totalLinhas)`.

4. **Uso de `\r` (carriage return) em vez de ANSI escape codes:** `\r` é suportado em todos os terminais Unix/Windows sem biblioteca extra. ANSI (`\033[2K\r`) apagaria a linha antes de reescrever, mas requereria detecção de suporte ANSI — complexidade desnecessária para v1. Aceito: possível artifact visual em terminais que não suportam `\r` (log files, IDEs) — mitigado pelo guard de TTY (AC9).

5. **Guard `System.console() == null` para detecção de TTY:** é o método canônico Java sem dependências externas. Limitação: retorna `null` em alguns IDEs mesmo com terminal interativo. Aceito para v1 — o pior caso é "não mostrar a barra no IDE", o que é aceitável.

6. **`resumo.erro()` em vez de `contadorErros[]`:** `ResumoExecucao` já expõe getter público `erro()` desde a Story 5.1 (verificado no fonte). Não há necessidade de array workaround `contadorErros[]` — `barra.atualizar(linhaAtual, resumo.erro())` reutiliza o contador já acumulado, sem duplicar estado. Isso elimina um parâmetro da assinatura de `processarLinha()` e evita inconsistência se os dois contadores divergirem.

7. **`linhaAtual = linhaExcel[0] - 1` dentro de `processarLinha`:** `linhaExcel[0]` é incrementado no início de `processarLinha()` (AC10 da Story 4.5 — conta cabeçalho como linha 1, dados a partir de 2). Portanto, ao chamar `barra.atualizar()` ao final da mesma função, `linhaExcel[0] - 1` representa o número de linhas de dados processadas até o momento. A barra chega a 100% quando `linhaAtual == totalLinhas`.

### Pontos verificados na arquitetura/PRD/AGENTS

- **NFR-08 (usabilidade):** não listado explicitamente no PRD do projeto, mas alinhado com a convenção de ferramentas CLI industriais e com o feedback do operador mencionado em `AGENTS.md` ("Nunca executa DML direto — operador monitora o processo").
- **`ExcelSessao.linhas()`** retorna `Stream<Map<String, String>>` lazy com cabeçalho já consumido. `count()` força a materialização de todo o stream — correto para contagem. [Source: `src/main/java/br/com/arxcode/tematica/geo/excel/ExcelSessao.java`]

## Out-of-scope desta story

- Cores ANSI na barra (verde/vermelho) — requer detecção de suporte ANSI; v2+.
- Estimativa de tempo restante (ETA) — requer taxa de linhas/s; v2+.
- Progresso em sub-fases (validação de conexão, carregamento do mapping) — fases rápidas; sem utilidade visual.
- Progresso dos outros subcomandos (`mapear`, `validar`) — `importar` é o único com iteração longa.
- Atualização parcial a cada N linhas configurável via `application.properties` — v2+.

## Trade-offs registrados

- **Segunda leitura do XLSX vs. barra sem total:** leitura dupla adiciona ~1–2 s mas entrega UX muito superior (porcentagem + ETA implícita). Para o tamanho de planilhas municipal (~1.000–5.000 linhas), aceitável.
- **`\r` vs. ANSI escape:** `\r` é mais simples e portável; ANSI seria mais robusto em terminais modernos. Aceito para v1; ANSI pode ser adicionado em v2 sem quebrar a interface de `BarraProgresso`.
- **Guard por `System.console()` vs. nenhum guard:** sem guard, a barra polui logs em CI/CD e redirecionamentos. Com guard, operadores em IDEs podem não ver a barra. Aceito: CI/CD é prioridade sobre conforto em IDE.
