## Contexto e Motivação

### O problema atual

`codigoImovel` e `sequenciaPredial` em `LinhaMapeada` são `String`. Cada consumidor precisa convertê-los ou coagi-los individualmente:

- `SqlGeradorUpdate` chama `coercionador.coagir(linha.codigoImovel(), Tipo.DECIMAL, null)` para gerar o `WHERE cadastrogeral = 123` — coerção redundante, pois a `String "123"` já deveria ter sido validada como numérica muito antes.
- `SqlGeradorUpsert` chama `coercionador.coagir(linha.codigoImovel(), Tipo.TEXTO, null)` — coerção de string como texto, tratando um identificador numérico como se fosse texto livre, o que gera `referencia = '12345'` (com aspas) em vez de `referencia = 12345` (numérico correto).
- `ImportarCommand` valida apenas se o valor está vazio, sem verificar se é um inteiro válido.

Resultado: erros de parsing para valores como `"ABC"` ou `"12.5"` chegam ao gerador de SQL em vez de serem detectados na leitura da planilha, gerando mensagens de erro confusas e longe da origem do dado.

### A solução

Converter `codigoImovel` e `sequenciaPredial` para `long` / `Long` logo após ler as células da planilha em `ImportarCommand.processarLinha`, **antes** de construir `LinhaMapeada`. Se a conversão falhar (valor não é inteiro), registrar no log como erro de linha (mensagem PT) e `continue` — pula a linha inteira. Alterar `LinhaMapeada` para carregar `long codigoImovel` e `Long sequenciaPredial`.

Com isso:
- `SqlGeradorUpdate` passa a receber valores já validados como `long` — usa `String.valueOf(linha.codigoImovel())` diretamente, sem coerção.
- `SqlGeradorUpsert` idem — usa `String.valueOf(linha.codigoImovel())` sem aspas, sem escape.
- Elimina o CAST `CAST(? AS numeric)` no SQL gerado.
- Detecta o erro na linha mais próxima da fonte: ao ler a planilha.

---

## User Story

> **Como** sistema de importação, **quero** converter `codigoImovel` e `sequenciaPredial` para `long`/`Long` ao ler cada linha da planilha, **para** que qualquer valor inválido (não-inteiro) seja detectado e registrado imediatamente no log, e para que os geradores de SQL recebam valores tipados corretamente — eliminando coerções redundantes e garantindo que `cadastrogeral` e `sequencia` apareçam sem aspas no SQL gerado.

## Dev Notes

### Motivação da conversão no `ImportarCommand`, não no `ExcelSessao`/`ExcelLeitor`

`ExcelSessao` retorna `Map<String, String>` propositalmente — é genérico e não conhece o domínio (qual coluna é `codigoImovel`). A conversão para `long` requer conhecimento do `Mapeamento` (qual header é a coluna-chave), que só está disponível no `ImportarCommand`. Portanto o local correto é `processarLinha`, logo após extrair a célula pelo header.

### Por que `long` (primitivo) para `codigoImovel`

`cadastrogeral` é `numeric(10,0)` e sempre positivo no domínio municipal. `long` (64 bits) comporta qualquer valor `numeric(10,0)` sem overflow (max 9.999.999.999 < `Long.MAX_VALUE`). Primitivo evita boxing/unboxing e null checks desnecessários.

### Por que `Long` (boxed) para `sequenciaPredial`

`sequenciaPredial` é `null` para TERRITORIAL — exige boxed. Para PREDIAL, o valor é sempre positivo.

### Impacto no `SqlGeradorUpdate` — simplificação real

Antes: `coercionador.coagir("12345", Tipo.DECIMAL, null)` → `ResultadoCoercao{literalSql="12345"}` — coerção que nunca pode falhar (um `long` parseado é sempre decimal válido).
Depois: `String.valueOf(12345L)` → `"12345"` diretamente, sem coercionador.

A coerção de `Tipo.DECIMAL` para `codigoImovel` e `sequenciaPredial` no `SqlGeradorUpdate` deixa de existir. Isso **simplifica** a assinatura de responsabilidade: o `SqlGeradorUpdate` não precisa mais saber que `codigoImovel` é numérico — ele simplesmente serializa o `long` que recebe.

### Impacto no `SqlGeradorUpsert` — correção de bug

Antes: `coercionador.coagir("12345", Tipo.TEXTO, null)` → literal `'12345'` (com aspas) → `referencia = '12345'`. Mas `referencia` é `numeric(10,0)` — isso só funciona por cast implícito do PostgreSQL e é semanticamente incorreto.
Depois: `String.valueOf(12345L)` → `"12345"` → `referencia = 12345`. Correto.

### Invariante do construtor de `LinhaMapeada` — AC1

Decisão fechada: `codigoImovel > 0` — alinhado ao domínio municipal onde `cadastrogeral` sempre é positivo. Mensagem PT: `"Código do imóvel deve ser maior que zero."`.

### Testes de `SqlGeradorUpdateTest` — mudança de fixture

Todas as instâncias de `new LinhaMapeada("123", null, ...)` passam a ser `new LinhaMapeada(123L, null, ...)`. Testes `codigoNaoNumerico_retornaFalha` e `predial_sequenciaNaoNumerica_retornaFalha` são removidos (essa responsabilidade migrou para `ImportarCommand`).

### Testes de `SqlGeradorUpsertTest` — mudança de fixture e SQLs esperados

Todas as instâncias de `new LinhaMapeada("R1", null, ...)` passam a usar valores numéricos reais (ex.: `new LinhaMapeada(12345L, null, ...)`). SQLs esperados mudam: `referencia = 'R1'` → `referencia = 12345`. Os `@Nested` `EscapeReferencia` e `EscapeInjection` são removidos.
