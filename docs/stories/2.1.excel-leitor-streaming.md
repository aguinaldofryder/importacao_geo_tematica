## User Story
> **Como** sistema, **quero** ler planilhas `.xlsx` em streaming, **para** processar arquivos com ≥ 100k linhas sem esgotar memória.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | `ExcelLeitor.abrir(Path xlsx)` retorna `ExcelSessao` (`AutoCloseable`); `close()` libera o `InputStream` e o `ReadableWorkbook`. |
| AC2 | `ExcelSessao.cabecalhos()` retorna `List<String>` com os valores da **primeira linha** da **aba ativa** (`ReadableWorkbook.getFirstSheet()`). |
| AC3 | `ExcelSessao.linhas()` retorna `Stream<Map<String,String>>`, onde cada `Map` mapeia `cabecalho → conteúdo da célula como String bruta` (sem coerção de tipo — coerção é responsabilidade da Story 4.1). |
| AC4 | Iteração é **streaming**: ao processar uma planilha sintética com 100.000 linhas × 4 colunas via `linhas().forEach(...)` em JVM forkada com `-Xmx256m`, o teste **conclui sem `OutOfMemoryError`**. Critério operacionalmente verificável via Surefire em fork dedicado (ver Testing); documenta a margem de 256 MB ≪ 512 MB do NFR-04. |
| AC5 | `Path` que não existe → lança `ImportacaoException` com mensagem em PT contendo o path informado (ex.: `"Arquivo não encontrado: /tmp/x.xlsx"`). |
| AC6 | Arquivo cuja extensão (case-insensitive) **não termina em `.xlsx`** → lança `ImportacaoException` com mensagem em PT (ex.: `"Formato não suportado (esperado .xlsx): planilha.xls"`). Esta é a única validação de formato — não há leitura de magic bytes; corrupção interna do XLSX é mapeada via `IOException` capturada no construtor da sessão e re-lançada como `ImportacaoException`. |
| AC7 | Planilha contendo apenas a linha de cabeçalho → `linhas()` retorna `Stream` vazio; `cabecalhos()` retorna a lista normalmente. |
| AC8 | Cabeçalhos duplicados ou vazios estão **fora de escopo** desta story: comportamento é "última coluna vence" (`Map.put` natural). Se aparecerem nas planilhas reais (Territorial/Predial), uma story específica de saneamento será aberta. |

## Dev Notes

- **API fastexcel-reader 0.18.4** (verificada contra README oficial em https://github.com/dhatim/fastexcel):
  ```java
  try (InputStream is = Files.newInputStream(path);
       ReadableWorkbook wb = new ReadableWorkbook(is)) {
      Sheet sheet = wb.getFirstSheet();
      try (Stream<Row> rows = sheet.openStream()) {
          rows.forEach(r -> { /* r.getCell(i).getRawValue() */ });
      }
  }
  ```
- **Coerção de valores:** todas as células são retornadas como `String` bruta via `Cell.getRawValue()`. Coerção de tipo (TEXTO/DECIMAL/DATA/MULTIPLA_ESCOLHA) é **responsabilidade da Story 4.1**.
- **Aba processada:** sempre `getFirstSheet()`. Múltiplas abas, `getActiveSheet()`, e seleção por nome estão fora de escopo (planilhas reais Territorial/Predial usam aba única).
- **Cabeçalho:** sempre a primeira linha (`row index 0` no fastexcel).
- **`ImportacaoException` é decisão arquitetural nova:** primeira story do projeto a definir hierarquia de exceções. Demais stories (4.4 log-erros-estruturado, 4.5 comando-importar, etc.) herdarão. Se @architect quiser registrar em `arquitetura.md` ou `CONVENCOES.md`, é apropriado — mas não bloqueia esta story.
- **Native-image (risco R1 do PRD):** `fastexcel-reader` usa Jackson e parsing XML internamente. Se prod-jar funcionar (T6), native-image provavelmente também funcionará; cobertura completa fica em Story 6.3 (e2e-binário-nativo). Se T6 falhar, registrar classes em `reflect-config.json`.
- **Out-of-scope desta story:**
  - Defesa contra zip-bomb (planilhas vêm de equipe interna conhecida).
  - Sanitização de paths (responsabilidade do chamador / comando CLI).
  - Cabeçalhos duplicados ou vazios (AC8).
  - Performance em CPU (apenas memória/streaming é NFR).

## Testing

- **Framework:** JUnit 5 + `@QuarkusTest` (padrão do projeto).
- **Fixtures:** geradas em runtime via `fastexcel-writer 0.20.0` em diretório `@TempDir`. Não versionar `.xlsx` em `src/test/resources` (planilhas são reproduzíveis e versão do writer fica em `pom.xml`).
- **Testes unit (`ExcelLeitorTest`)** — devem rodar em `mvn test` (Surefire), execução rápida:
  - Happy path: 3 colunas × 5 linhas, valida `cabecalhos()` e ordem dos `Map`s.
  - AC7 vazio: só header → `linhas().count() == 0`.
  - AC5 inexistente: `assertThrows(ImportacaoException.class, ...)`, mensagem contém path.
  - AC6 extensão errada: `.txt` e `.xls` → ambos `ImportacaoException`.
  - AC6 corrompido: arquivo `.xlsx` com bytes inválidos → `ImportacaoException` com `cause` do tipo `IOException`.
- **Teste streaming (`ExcelLeitorStreamingTest`)** — roda em **Surefire** (`mvn test` e `mvn verify`); **não usar sufixo `IT`** (skipITs=true global no pom). Heap controlado via `-Xmx256m`:
  - Opção recomendada: aplicar `-Xmx256m` no `<argLine>` do `maven-surefire-plugin` no `pom.xml` (afeta todos os testes; aceitável pois nenhum outro teste atual exige mais). @dev decide e documenta.
  - Opção alternativa: `surefire.forkNode` dedicado para classe taggeada `@Tag("streaming")`.
  - Gerar 100k linhas × 4 colunas em `@TempDir` via `fastexcel-writer`.
  - Iterar com `linhas().forEach(...)` contando linhas; assertar `count == 100_000`.
  - Sucesso = não estourar OOM; medição fina de pico de memória **não é necessária** — o teste é binário (passou ou OOM).
