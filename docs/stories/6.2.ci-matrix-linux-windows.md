## User Story
> **Como** equipe, **quero** CI que compile o binário nativo em Linux **e** Windows em paralelo, **para** entregar dois binários prontos para o operador a cada release.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | Workflow `.github/workflows/native-build.yml` dispara em push para `main` e em qualquer tag `v*.*.*`; **não** dispara em pull_request (build nativo é lento — PR usa `ci.yml` JVM) |
| AC2 | Matrix strategy com dois runners: `ubuntu-latest` e `windows-latest`; cada job executa `./mvnw package -Pnative -DskipTests` de forma independente |
| AC3 | Cache de `~/.m2` (chave baseada em `pom.xml` hash) e da instalação GraalVM configurados via `actions/cache` ou opções do `graalvm/setup-graalvm@v1` |
| AC4 | Artefato Linux nomeado `importacao-geo-linux-x64` (binário sem extensão) e artefato Windows nomeado `importacao-geo-windows-x64.exe`; upload via `actions/upload-artifact@v4`; retenção de 30 dias |
| AC5 | Job falha explicitamente se binário > 100 MB: checagem com script antes do upload (Linux: `wc -c`; Windows: PowerShell `(Get-Item ...).length`) |
| AC6 | Tempo total por job < 15 min (target; documentar no Dev Notes estratégia de cache para atingir) |
| AC7 | Smoke test inline pós-build: executa `<binario> --help` e valida exit code 0; falha o job se exit code ≠ 0 |

## Dev Notes

### GraalVM — versão fixada
Usar o mesmo major/minor da Story 6.1 (documentado em `DEVELOPING.md`). Exemplo de configuração:
```yaml
- uses: graalvm/setup-graalvm@v1
  with:
    java-version: '21'
    distribution: 'graalvm'
    github-token: ${{ secrets.GITHUB_TOKEN }}
    cache: 'maven'          # habilita cache nativo do setup-graalvm para ~/.m2
    native-image-job-reports: 'true'
```
Se `cache: 'maven'` cobrir `~/.m2`, remover step separado de `actions/cache` para evitar conflito.

### Caminho do binário por OS
O Quarkus native gera:
- **Linux:** `target/<finalName>-runner` (sem extensão)
- **Windows:** `target/<finalName>-runner.exe`

Usar matrix variable para abstrair:
```yaml
strategy:
  matrix:
    include:
      - os: ubuntu-latest
        binary: target/importacao-geo-runner
        artifact: importacao-geo-linux-x64
      - os: windows-latest
        binary: target/importacao-geo-runner.exe
        artifact: importacao-geo-windows-x64.exe
```

### Windows — MSVC Build Tools
`graalvm/setup-graalvm@v1` configura MSVC automaticamente em `windows-latest` via `vcvars64.bat`. Não é necessário step adicional. Se o build falhar com erro de compilador C, checar se `windows-latest` ainda inclui Visual Studio Build Tools 2022.

### Checagem de tamanho — scripts por OS
```yaml
# Linux
- name: Verificar tamanho do binário
  run: |
    SIZE=$(wc -c < ${{ matrix.binary }})
    echo "Tamanho: ${SIZE} bytes"
    [ "$SIZE" -le 104857600 ] || (echo "ERRO: binário excede 100 MB" && exit 1)

# Windows (PowerShell)
- name: Verificar tamanho do binário
  shell: pwsh
  run: |
    $size = (Get-Item "${{ matrix.binary }}").length
    Write-Host "Tamanho: $size bytes"
    if ($size -gt 104857600) { throw "ERRO: binário excede 100 MB" }
```
Usar `if: runner.os == 'Linux'` / `if: runner.os == 'Windows'` para separar os steps se necessário, ou encapsular em script condicional com `${{ runner.os }}`.

### Cache e tempo de build
- Primeira execução sem cache: ~20-25 min (GraalVM download + compilação native). Após cache aquecido: ~10-12 min.
- AC6 (< 15 min) aplica-se com cache quente. Documentar isso no comentário do workflow.
- A chave de cache deve incluir `runner.os` para evitar cross-contamination entre Linux e Windows.

### Relacionamento com `ci.yml`
O `ci.yml` existente (JVM mode, `./mvnw -B verify`) continua inalterado e roda em PRs. `native-build.yml` é complementar e roda apenas em push/main e tags — builds nativos não devem bloquear o fluxo de review de PRs.
