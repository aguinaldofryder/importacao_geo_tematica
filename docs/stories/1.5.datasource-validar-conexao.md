## User Story
> **Como** operador, **quero** validar a conexão ao banco antes de qualquer operação, **para** falhar-rápido se a configuração estiver errada.

**Implementação do comando:**
- Injetar `AgroalDataSource` (ou `javax.sql.DataSource`) via `@Inject`.
- Usar `DataSource.getConnection()` em try-with-resources.
- Extrair metadados: `connection.getMetaData()` → `.getURL()`, `.getUserName()`, `.getDatabaseProductVersion()`.
- Parse do host:port a partir da URL JDBC (ex: regex `jdbc:postgresql://([^/]+)/(.+)`).

**Segurança (AC5):** ao formatar mensagem de erro de `PSQLException`, nunca incluir `e.getMessage()` diretamente se puder conter a senha. Usar apenas a mensagem mapeada da tabela abaixo.

**Tabela de mapeamento PSQLException → mensagem PT:**
| SQLState | Situação | Mensagem PT |
|---|---|---|
| `08001` | Servidor inatingível | `Servidor indisponível: verifique host e porta` |
| `08006` | Conexão interrompida | `Conexão encerrada inesperadamente` |
| `28P01` | Credencial inválida | `Usuário ou senha incorretos` |
| `3D000` | Base não existe | `Base de dados não encontrada` |
| `57014` | Query cancelada (timeout) | `Tempo limite de conexão esgotado` |
| outros | Genérico | `Falha na conexão: <SQLState>` |

**Testcontainers:**
```java
@Container
static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16-alpine");
```
- Para cenário de timeout: usar `pg.withStartupTimeout(...)` ou container com porta errada.
- `%test.quarkus.datasource.active=true` no profile de teste (diferente da 1.3 que usava `false`).
- Sobrescrever URL/user/pass via `@QuarkusTestResource` ou `quarkus.datasource.jdbc.url=${pg.getJdbcUrl()}`.

### Refinamento Técnico — @architect

**1. `Callable<Integer>` obrigatório (AC7)**

O stub atual implementa `Runnable` — exit code é sempre 0. Trocar para `Callable<Integer>`:

```java
@ApplicationScoped
@CommandLine.Command(name = "validar-conexao", ...)
public class ValidarConexaoCommand implements Callable<Integer> {

    @Inject
    DataSource dataSource;          // javax.sql.DataSource via quarkus-agroal

    @Override
    public Integer call() {
        // retorna 0 ou 1
    }
}
```

`@ApplicationScoped` é obrigatório — sem ele o CDI não injeta `DataSource` no subcomando (o `IFactory` do picocli resolve beans CDI, mas a classe precisa ser um bean).

**2. Injeção de DataSource**

Com um único datasource configurado, `@Inject DataSource` (javax.sql.DataSource) é suficiente. Não usar `@io.quarkus.agroal.DataSource` qualifier (qualifier só é necessário quando há múltiplos datasources).

**3. Estratégia de testes — não quebrar os existentes**

`application.properties` já tem:
```properties
quarkus.devservices.enabled=false
```
Os testes da story 1.3 (`ValidarConexaoCommandTest`, etc.) rodavam com datasource inativo. Para story 1.5, os novos testes precisam de datasource ativo via Testcontainers. Estratégia sem conflito:

- Criar `ValidarConexaoIntegrationTest` separado com `@TestProfile` próprio
- O `TestProfile` habilita datasource e injeta a URL do container:

```java
public class PostgresTestProfile implements QuarkusTestProfile {
    @Override
    public Map<String, String> getConfigOverrides() {
        // URL será sobrescrita pelo QuarkusTestResourceLifecycleManager
        return Map.of("quarkus.datasource.active", "true");
    }
}
```

- Usar `QuarkusTestResourceLifecycleManager` para iniciar o container e injetar a URL:

```java
public class PostgresResource implements QuarkusTestResourceLifecycleManager {
    static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16-alpine");

    @Override
    public Map<String, String> start() {
        pg.start();
        return Map.of(
            "quarkus.datasource.jdbc.url",      pg.getJdbcUrl(),
            "quarkus.datasource.username",       pg.getUsername(),
            "quarkus.datasource.password",       pg.getPassword(),
            "quarkus.datasource.jdbc.acquisition-timeout", "5"
        );
    }

    @Override
    public void stop() { pg.stop(); }
}
```

**4. Schema `aise` não existe no Testcontainers**

`application.properties` em dev tem `currentSchema=aise`. O container Testcontainers cria um PostgreSQL limpo sem schema `aise`. O `QuarkusTestResourceLifecycleManager` NÃO deve incluir `currentSchema` nas overrides — `SELECT 1` não usa schema, portanto o teste de conexão funciona sem ele.

**5. Cenário de timeout — abordagem pragmática**

`acquisition-timeout` do Agroal age sobre a fila de conexões, não sobre o TCP. Para simular timeout de forma confiável:
- Usar um container que inicia **mas recusa conexões** via `pg_hba.conf` customizado, OU
- Abordagem mais simples: definir `acquisition-timeout=1` (1 segundo) e usar `DataSource` com pool de 0 conexões disponíveis (sem container real) — garante timeout na aquisição

No `PostgresResource`, adicionar variante de teste de timeout:
```java
// No cenário de timeout: sobrescrever apenas o acquisition-timeout e usar host inválido
"quarkus.datasource.jdbc.url" → "jdbc:postgresql://127.0.0.1:9999/noexist"
"quarkus.datasource.jdbc.acquisition-timeout" → "1"
```
Isso gera SQLState `08001` (servidor inatingível) dentro do timeout configurado.

**6. GraalVM native-image**

`javax.sql.DataSource` + pgjdbc + `getMetaData()` têm suporte nativo completo no Quarkus 3.x — nenhuma configuração adicional de reflexão necessária para esta story.
