## User Story
> **Como** sistema, **quero** converter valores de célula (string bruta) em literais SQL tipados corretos, **para** gerar SQL válido e seguro contra injection.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | `Coercionador.coagir(String valor, Tipo tipo, Map<String,Long> alternativas)` retorna `Resultado(Object valorLiteral, boolean ok, String erro)` |
| AC2 | `TEXTO`: trim, escape `'` → `''`, retorna literal entre aspas simples (`'texto'`). Nulo/vazio → literal `NULL` |
| AC3 | `DECIMAL`: aceita `"1.234,56"` (pt-BR) e `"1234.56"` com normalização de locale (vírgula → ponto). Invalid → `ok=false, erro="valor X não é decimal válido"` |
| AC4 | `DATA`: aceita `dd/MM/yyyy` e `yyyy-MM-dd`; retorna literal `DATE 'yyyy-MM-dd'`. Invalid → `ok=false` |
| AC5 | `MULTIPLA_ESCOLHA`: resolve string → id via `alternativas` (do mapping.json). Sem match → `ok=false, erro="alternativa 'X' não mapeada"` (este caso não deve ocorrer em runtime — validador bloqueia antes) |
| AC6 | Valor null/vazio em qualquer tipo → retorna literal `NULL` (ok=true) |
| AC7 | **Cobertura ≥ 80%** neste módulo (crítico para NFR-09) |
| AC8 | Testes unit exaustivos: happy paths + todas as falhas + edge cases (ponto sem decimais, data com ano de 2 dígitos rejeitada, texto com aspas simples) |

## Dev Notes

- Gerador de SQL usa **literais** (não PreparedStatement) porque produz arquivo texto para o DBA aplicar.
- Segurança: escape correto é responsabilidade desta classe.
- Formatos de DATA configuráveis no futuro; v1 aceita apenas os dois formatos acima.
- Para DECIMAL: tratar também notação científica negada; rejeitar valores com caracteres estranhos.

## Testing
- **Módulo crítico** — cobertura obrigatória ≥ 80%.
- Testes de segurança: `"O'Brien"` vira `'O''Brien'`.
