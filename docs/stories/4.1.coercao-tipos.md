## User Story

> **Como** sistema gerador de SQL, **quero** converter strings cruas vindas do Excel em **literais SQL tipados** (texto entre aspas com escape, decimais com ponto, datas em ISO `DATE 'yyyy-MM-dd'`, ids inteiros para multipla-escolha), **para** que os arquivos `.sql` produzidos sejam (a) sintaticamente válidos no PostgreSQL, (b) seguros contra SQL injection — já que o pipeline **não** usa `PreparedStatement` (CON-03), e (c) reportem erros de domínio (valor inválido) por linha sem abortar a execução.

## Acceptance Criteria

| # | Critério |
|---|---|
| AC1 | Existe `Coercionador` em `br.com.arxcode.tematica.geo.geracao` com método público `ResultadoCoercao coagir(String valor, Tipo tipo, Map<String,Integer> alternativas)`. `alternativas` é o mesmo `Map` exposto por `ColunaDinamica.alternativas()` (Story 3.1) — `null` ou vazio é aceito para tipos `≠ MULTIPLA_ESCOLHA`. |
| AC2 | Existe `record ResultadoCoercao(String literalSql, boolean ok, String erro)` em `br.com.arxcode.tematica.geo.geracao` com fábricas estáticas `ok(String literal)` e `falha(String mensagemPt)`. Em sucesso: `ok=true`, `erro=null`. Em falha: `ok=false`, `literalSql=null`. Construtor canônico valida invariantes (`ok=true` exige `literalSql != null`; `ok=false` exige `erro != null`). |
| AC3 | **Valor nulo/vazio (qualquer tipo)** — `valor==null` ou `valor.trim().isEmpty()` retorna `ResultadoCoercao.ok("NULL")` para qualquer `tipo`. Não há diferenciação entre "vazio" e "null" no Excel (fastexcel devolve string vazia para células em branco — Story 2.1). |
| AC4 | **`TEXTO`** — aplica `trim`, escapa aspas simples (`'` → `''`), envolve em aspas simples. Exemplos: `"O'Brien"` → `'O''Brien'`; `"  texto  "` → `'texto'`; `"a\nb"` → `'a\nb'` (newline literal preservada — PostgreSQL aceita; `E''` não é necessário). Backslash **não** é escapado (no PostgreSQL padrão, `standard_conforming_strings=on`, backslash em string literal não-`E` é literal). **Nota de precedência:** o guard de AC3 (null/trim-vazio → `NULL`) **precede** este branch. Strings com algum conteúdo após `trim` chegam aqui; `"   "` (só espaços) já virou `NULL` em AC3 e nunca atinge a lógica de TEXTO. |
| AC5 | **`DECIMAL`** — aceita formato pt-BR `"1.234,56"` (ponto como separador de milhar, vírgula decimal) **ou** formato ISO `"1234.56"` (sem milhar, ponto decimal). Heurística: se a string contém `,`, trata `.` como milhar e `,` como decimal; senão, `.` é decimal. Resultado normalizado para literal SQL `1234.56` (sem aspas). Aceita sinal negativo (`-1.234,56` → `-1234.56`) e ausência de decimais (`"42"` → `42`). **Rejeita**: notação científica (`1e3`), letras, múltiplas vírgulas/pontos decimais ambíguos, espaços internos. Falha com mensagem PT: `"valor 'X' não é decimal válido"`. |
| AC6 | **`DATA`** — aceita os formatos `dd/MM/yyyy` (pt-BR) e `yyyy-MM-dd` (ISO). Resultado: literal SQL `DATE 'yyyy-MM-dd'` (sempre normalizado para ISO). Validação **estrita** via `DateTimeFormatter` com `ResolverStyle.STRICT` — `31/02/2025` e `2025-02-31` são rejeitados. Ano deve ter 4 dígitos (rejeita `01/01/25`). Falha: `"valor 'X' não é data válida (formatos aceitos: dd/MM/yyyy, yyyy-MM-dd)"`. |
| AC7 | **`MULTIPLA_ESCOLHA`** — resolve `valor` (após `trim`) consultando o `Map<String,Integer> alternativas` recebido. Match é case-insensitive **e** insensível a espaços extras (consistente com `importacao.mapeamento.case-sensitive=false` e `trim-espacos=true` da Story 1.4 / `ImportacaoConfig`). Sucesso → literal SQL com o id (`"42"`, sem aspas — coluna `idalternativa` é numérica). Sem match → `falha("alternativa 'X' não mapeada")`. Em runtime normal este caminho **não deve ocorrer** (Story 3.3 bloqueia execução com qualquer item PENDENTE), mas a falha defensiva é obrigatória. Se `alternativas` for `null`/vazio, retorna `falha("mapeamento ausente para tipo MULTIPLA_ESCOLHA")` — programação defeituosa, não dado ruim. **Defesa em profundidade — value=null:** se o match encontrar uma chave cujo `value==null` no `Map` (cenário possível porque `AutoMapeador.java:213` insere `put(valor, null)` para alternativas não resolvidas, e a Story 3.3 hoje valida apenas estrutura `MAPEADO`), retornar `falha("alternativa 'X' encontrada no mapeamento mas com id nulo (mapeamento corrompido)")`. Não deve ocorrer pós-validação da 3.3 (que rejeita `ColunaDinamica` MAPEADO com alternativas inconsistentes), mas a defesa evita NPE silencioso se essa invariante regredir. |
| AC8 | **Imutabilidade e thread-safety** — `Coercionador` não tem estado mutável (sem campos não-final, sem caches mutáveis). Pode ser instanciado uma vez e compartilhado. **Não** é `@ApplicationScoped` nesta story — o consumidor (Stories 4.2/4.3) decide o ciclo de vida; o construtor é público sem argumentos. |
| AC9 | **Segurança (CON-03)** — testes dedicados verificam que: (a) `"O'Brien"` produz `'O''Brien'`; (b) `"'; DROP TABLE x; --"` produz `''';' DROP TABLE x; --'` (texto inerte); (c) tentativa de injection em campo `MULTIPLA_ESCOLHA` (string `"1; DROP"`) cai no caminho `alternativa 'X' não mapeada` e **não** vira literal numérico. |
| AC10 | **Cobertura JaCoCo ≥ 80 % linhas** na classe `Coercionador` (módulo crítico NFR-09 — meta global é 70 %, esta story atinge folga). `ResultadoCoercao` 100 % (record com 2 fábricas). Verificação por inspeção de `target/site/jacoco/...` ou execução local de `./mvnw verify`. |
| AC11 | **Mensagens 100 % em PT** (NFR-01) — nenhuma string voltada ao log/usuário em inglês. `Logger` (`org.jboss.logging`) **não** é usado nesta classe — coerção é função pura, falha é devolvida via `ResultadoCoercao.erro` (quem loga é o `SqlGerador`/`ImportarCommand`). |
| AC12 | **`mvn verify` verde**. Nenhuma dependência nova no `pom.xml` — `BigDecimal`, `LocalDate`, `DateTimeFormatter` são JDK; `Tipo` já existe (Story 2.2). |

## Dev Notes

### Decisões arquiteturais

- **Função pura, sem CDI** — `Coercionador` não usa `@Inject` nem `@ApplicationScoped`. Razões: (a) função determinística sem efeitos colaterais; (b) testabilidade trivial (JUnit puro, sem `@QuarkusTest`); (c) consumidores (4.2/4.3) ficam livres para instanciar uma vez ou compartilhar — decisão deles. Padrão idêntico ao da Story 3.1 (`MapeamentoStore` recebe `ObjectMapper` no construtor) e 2.2 (records de domínio sem CDI).
- **`ResultadoCoercao` em vez de exceção** — coerção falha **por linha** e o pipeline precisa continuar (PRD FR-13: "registrando erro no `.log` em falhas"). Exceção para fluxo de controle por linha seria caro e ruidoso. `Optional<String>` foi descartado porque perderia a mensagem PT específica para o log.
- **Locale fixo (pt-BR ↔ ISO) em vez de `Locale` dinâmico** — escopo do projeto é IPTU municipal brasileiro; planilhas são pt-BR e o banco IPTU consome ISO. Tornar configurável é over-engineering para v1. Documentar em `Coercionador` Javadoc.
- **`ResolverStyle.STRICT` + `uuuu`** — `DateTimeFormatter` em modo SMART (default) **aceita** `31/02/2025` (clamp para 28/02). Para garantir AC6 (rejeitar datas inexistentes), STRICT é obrigatório; e STRICT exige `uuuu` (year-of-era ambíguo com `yyyy` em strict). Detalhe sutil mas crítico — sem isso, AC6 falha silenciosamente.
- **Newline / backslash em TEXTO sem flag `E''`** — PostgreSQL com `standard_conforming_strings=on` (default desde 9.1) trata backslash como literal em strings `'...'` normais. Newline literal dentro de string com aspas simples também é aceita. Não precisamos do prefixo `E` (escape string). Se a configuração do PostgreSQL alvo divergir, esta classe precisará revisão — registrar como risco em out-of-scope.
- **`MULTIPLA_ESCOLHA` com normalização fixa** — case-insensitive + trim alinha com `ImportacaoConfig.mapeamento` defaults (`caseSensitive=false`, `trimEspacos=true`). Ler `ImportacaoConfig` aqui acoplaria a classe ao CDI sem ganho real (Story 3.2 já normaliza no momento do auto-mapeamento; o `Map` chega "limpo"). Match insensitive aqui é cinto-e-suspensórios para edge cases (planilhas com case alterado entre `mapear` e `importar`).
- **Caminho `alternativa não mapeada` "não deve ocorrer em runtime"** — Story 3.3 (`MapeamentoValidador`) bloqueia execução do `importar` enquanto houver `PENDENTE`. Mas o caminho é mantido como defensivo — se alguém edita o `.json` e remove a entrada manualmente, queremos falha clara, não NPE.

### Trade-offs registrados

- **Regex `^-?\d+(\.\d+)?$` para DECIMAL** rejeita formatos exóticos mas válidos (ex.: `+1.5` com sinal positivo explícito, `1.000.` com ponto trailing). Decisão: rejeitar — planilhas reais não usam essas grafias; em caso contrário aparece no `.log` e o operador corrige a planilha.
- **Match case-insensitive em `MULTIPLA_ESCOLHA`** pode esconder dois valores realmente distintos do banco que diferem só por case. Risco baixo: catálogo `alternativa` do IPTU não tem entradas que diferem só por case dentro do mesmo `idcampo` (verificável via SQL ad-hoc se houver dúvida).
